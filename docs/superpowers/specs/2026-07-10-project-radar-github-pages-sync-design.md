# Project Radar GitHub Pages 定时同步设计说明

> 状态：已确认
>
> 日期：2026-07-10
>
> 受众：文档站维护者与本机同步任务维护者

## 1. 背景与目标

现有 Project Radar 文档站已经能够把 `/Users/baowenzhuo/project/xhxagentv3/docs/bwz` 渲染为静态快照，但定时发布依赖 Codex Agent，公开的 `chatgpt.site` 地址还会对当前电脑返回 Cloudflare 403。新方案将托管迁移到独立的公开 GitHub Pages 仓库，并使用固定代码每 30 分钟检查一次来源目录。

目标如下：

- 定时检查、快照生成、提交和触发发布均不依赖 Agent。
- AgentV3 仓库始终是只读内容来源，不产生站点文件、提交或配置。
- 内容未变化时不提交、不构建、不发布。
- 内容变化后，在一次定时周期内推送，并由 GitHub Actions 自动发布。
- 新站点不再使用 `chatgpt.site` 域名。
- 同步失败可诊断，并在下一个周期自动重试。

## 2. 范围与约束

### 2.1 包含范围

- 独立公开仓库 `1firecracker/project-radar-docs`。
- GitHub Pages 静态站点及 GitHub Actions 发布工作流。
- 本机固定 Node.js 同步脚本。
- macOS 用户级 LaunchAgent，每 30 分钟触发，并在登录后执行一次。
- Markdown、HTML、图片和附件的静态快照。
- 内容变化检测、并发锁、日志、失败重试和部署验证。
- 旧 Codex 定时任务和旧实时监听器的停用。

### 2.2 不包含范围

- 在线编辑、评论、搜索、访问控制和多内容来源。
- 每 30 分钟强制产生一次空提交或无内容变化的部署。
- 继续维护 Sites 的 R2 同步接口或 `/api/sync` 写入链路。
- 将 GitHub Pages 文件、配置或 Git 历史写入 AgentV3 仓库。

### 2.3 仓库隔离

- `/Users/baowenzhuo/project/xhxagentv3` 只允许读取 `docs/bwz`，同步脚本不得在该仓库内执行写操作、暂存、提交、分支切换或生成文件。
- 站点源码、快照、测试、工作流、日志和本机同步脚本只存在于独立站点项目。
- GitHub 远端只接收独立站点仓库的提交，不包含 AgentV3 的源码或 Git 历史。
- 运行时锁和日志写入独立站点项目的 `work/` 或用户 Application Support 目录，并由站点仓库忽略。

## 3. 方案选择

采用“本机固定脚本推送源码，GitHub Actions 构建并发布 Pages”的方案。

未选择的方案：

- 本机直接构建并推送 `dist`：实现较少，但构建产物污染提交历史，失败记录和回滚能力较弱。
- 继续通过 Sites 发布：保存版本和发布仍依赖 Sites 管理接口及短期凭证，无法满足完全脱离 Agent 的目标。

## 4. 总体架构

```text
/Users/baowenzhuo/project/xhxagentv3/docs/bwz
                         │ 只读扫描
                         ▼
          macOS LaunchAgent（每 30 分钟）
                         │
                         ▼
       固定 Node.js 同步脚本 + 互斥锁 + 日志
                         │ 内容有变化才提交和推送
                         ▼
       1firecracker/project-radar-docs（公开仓库）
                         │ push main
                         ▼
          GitHub Actions：测试 → 构建 → Pages 发布
                         │
                         ▼
       https://1firecracker.github.io/project-radar-docs/
```

GitHub Pages 发布由固定工作流完成。Agent 只用于首次搭建和人工维护，不参与后续定时同步。

## 5. 组件设计

### 5.1 静态快照生成器

沿用内容寻址快照：

- 递归扫描来源目录中的普通文件，拒绝隐藏路径和符号链接。
- 为每个文件计算 SHA-256、字节数、媒体类型和类别。
- 清单按路径稳定排序，内容身份相同则修订号相同。
- 生成 `public/content/manifest.json`、`public/content/objects/<sha256>` 和 `public/content/raw/<relative-path>`。
- 只有内容身份变化或快照缺失时才改写输出。

页面将原“同步时间”文案改为“内容更新时间”。该时间表示当前内容版本的生成时间，而不是最近一次定时检查时间。无变化检查不会伪造新的内容更新时间。

### 5.2 GitHub Pages 静态应用

现有 Sites 应用包含服务端 Worker 路径，不能原样部署到 GitHub Pages。迁移后提供纯静态 React 构建：

- 直接读取同源的 `content/manifest.json` 和内容对象。
- 复用现有导航、Markdown 安全渲染、HTML 沙箱和样式。
- 使用 Hash 路由承载文档深层链接，避免 GitHub Pages 对客户端路径返回 404。
- 所有资源路径基于仓库 Pages 基路径 `/project-radar-docs/` 生成。
- 首页、图片、附件和中文路径在该基路径下可直接访问。

旧 Sites Worker、R2 和同步接口不进入 GitHub Pages 运行时；迁移完成后可保留在历史提交中，但不再参与生产链路。

### 5.3 固定同步脚本

同步脚本一次运行的顺序如下：

1. 在站点项目的 `work/` 中获取互斥锁；已有任务运行时安全退出。
2. 记录 AgentV3 仓库的 Git 状态摘要，仅用于前后只读校验。
3. 运行快照生成器并读取结构化结果。
4. 若 `changed=false`，记录“内容已是最新”并退出，不提交或推送。
5. 若 `changed=true`，运行单元测试、同步测试、静态构建、渲染测试和 `git diff --check`。
6. 再次确认 AgentV3 Git 状态与运行前一致。
7. 只暂存生成快照及本次必要的站点文件，创建 `docs: refresh Project Radar snapshot` 提交。
8. 通过 SSH 推送独立站点仓库的 `main` 分支。
9. 写入成功结果并释放锁。

脚本不调用 Codex、Sites 管理接口、`/api/sync` 或任何 Agent 命令。

### 5.4 macOS LaunchAgent

用户级 LaunchAgent 使用固定配置：

- `RunAtLoad=true`，登录或重新加载任务后立即检查一次。
- `StartInterval=1800`，每 30 分钟触发一次。
- Node.js 和站点项目路径使用绝对路径，避免图形会话 PATH 不完整。
- 标准输出和错误日志写入独立站点项目的忽略目录。
- 不配置 `KeepAlive`，避免失败时形成高频重启循环。

Mac 关机或休眠期间不会执行本地扫描；恢复后通过登录启动或下一个 30 分钟周期补检查。内容变化不会丢失，因为每次都进行完整扫描。

### 5.5 GitHub 身份验证

- 首次创建仓库和启用 Pages 可使用已安装的 GitHub 连接器。
- LaunchAgent 不依赖连接器；它使用本机 SSH Agent 或钥匙串中已授权的 GitHub SSH 密钥推送。
- 私钥、访问令牌和连接器凭证均不写入仓库、LaunchAgent 正文或日志。
- 实施前必须验证非交互式 `git push` 能成功；无法验证时不启用定时任务。

### 5.6 GitHub Actions 发布

`main` 分支收到提交后，固定工作流执行：

1. 检出指定提交并安装锁定依赖。
2. 运行单元测试、同步测试和静态构建验证。
3. 上传静态产物为 GitHub Pages artifact。
4. 使用 GitHub Pages 官方发布动作部署该 artifact。
5. 在 GitHub 中保留运行状态和失败日志。

工作流只授予读取仓库、写入 Pages 和取得部署身份所需的最小权限。并发策略取消同一分支的旧部署，确保最终发布最新提交。

## 6. 一致性与时间语义

- 文件内容集合决定快照修订号；相同内容不会产生新修订。
- Git 提交与 GitHub Pages 部署均以完整快照为单位，读者不会看到半生成状态。
- 页面显示“内容更新时间”，只在内容变化并生成新清单时更新。
- “最近检查时间”和运行结果属于本机运维日志，不冒充内容更新时间。
- GitHub Actions 发布失败时，线上继续保留上一成功版本；下次内容变化或人工重跑可恢复。

## 7. 故障处理

| 场景 | 处理方式 |
|---|---|
| 内容无变化 | 记录成功检查，不创建提交或部署 |
| 上一次任务仍运行 | 新任务检测到锁后退出，避免并发提交 |
| Mac 休眠或关机 | 恢复后执行完整扫描，补同步全部变化 |
| 测试或构建失败 | 不提交、不推送，记录失败，下周期重试 |
| GitHub 暂时不可达 | 保留本地快照和提交状态，下周期检测后重试推送 |
| SSH 凭证失效 | 明确记录认证失败，不回退到明文令牌 |
| GitHub Actions 失败 | 保留上一线上版本，通过 Actions 日志诊断 |
| AgentV3 状态前后不同 | 中止提交并记录隔离校验失败 |
| 来源文件在扫描时变化 | 快照生成结束前复核内容身份，必要时重新扫描 |

同步脚本需要区分“快照已生成但未提交”“已有本地提交但尚未推送”和“远端已包含当前修订”三种恢复状态，保证网络失败后不会丢失更新或制造重复提交。

## 8. 测试与验收

### 8.1 单元和集成测试

- 无变化时不修改清单、不提交、不推送。
- 新增、修改、重命名和删除均产生正确快照。
- 隐藏路径、符号链接和路径穿越被拒绝。
- 互斥锁阻止重叠运行，过期锁可安全恢复。
- 测试失败、构建失败和推送失败不会发布不完整内容。
- 已提交未推送状态能在下次运行恢复。
- AgentV3 工作树在成功和失败路径中均保持不变。

### 8.2 GitHub Pages 验收

- 仓库公开且 Pages 来源为 GitHub Actions。
- 首页、Hash 深层链接、中文文档路径、图片和附件正常。
- Markdown 原始 HTML 不执行，原生 HTML 保持受限沙箱。
- 页面显示“内容更新时间”，不再误称无变化检查时间。
- 修改临时来源副本后，固定脚本能触发一次完整发布。
- 正式来源目录发生变化后，30 分钟内生成提交，GitHub Actions 成功后新内容可见。
- 当前电脑和另一台电脑都能打开新的 GitHub Pages 地址。

## 9. 迁移与回退

迁移顺序：

1. 在独立仓库完成静态 Pages 构建、测试和预览。
2. 创建公开 GitHub 仓库，配置 Pages 和 SSH 推送。
3. 手动执行一次固定同步脚本并验证生产地址。
4. 安装并加载 LaunchAgent。
5. 确认定时任务至少成功执行一个周期。
6. 删除 Codex 定时任务，确认旧实时监听器仍保持卸载。

在 GitHub Pages 验收前保留现有 Sites 站点，不删除线上内容。若迁移失败，停止新的 LaunchAgent 即可回退到已发布的 Sites 静态版本；AgentV3 不受影响。

## 10. 完成标准

以下条件全部满足时完成迁移：

1. `1firecracker/project-radar-docs` 为独立公开仓库。
2. GitHub Pages 生产地址可访问并完整呈现当前文档。
3. 本机固定同步脚本不调用 Agent，内容变化后能提交和推送。
4. LaunchAgent 每 30 分钟触发，登录后立即检查一次。
5. GitHub Actions 自动测试、构建和发布成功。
6. Codex 定时任务已删除，旧实时监听器保持卸载。
7. AgentV3 仓库前后状态完全一致，且没有新增站点文件、提交或配置。
8. 密钥、令牌和 SSH 私钥未出现在仓库、日志或回复中。
