## 1. 背景与目标

用户可能在一次工作后离开几天或一个周末，再回到页面时已经不清楚：上次完成了什么、还剩什么、应该从哪里继续。

恢复简报（Resume Brief）是一项只读恢复能力：用户重新进入时，基于已有 Project、Item 和 Event 快速说明当前进度和恢复入口。

它解决的是“状态断裂”，不是 Radar 的“推进中断”：

- Resume Brief 回答“上次做到哪里，现在如何恢复”。
- Radar 回答“现在是否有一步是用户还没有提出、但已经可以推进的”。

## 2. 能力边界

| 能力 | 触发 | 主要输出 | 是否改变状态 |
|---|---|---|---|
| Organizer | 完整 QA 结束 | Event、Item / Project 状态更新 | 是 |
| Radar | Post-run 状态变化后 | 一条可执行主动建议 | 用户授权后是 |
| Resume Brief | Re-entry（隔日或长时间返回） | 只读进度简报与恢复入口 | 否 |

Resume Brief 不调用 Organizer、不创建 Event、不新生 Radar，也不替用户执行任务。用户点击“继续”并发送新问题后，才进入正常的 Post-run 链路。

## 3. 触发条件

V0.2 初版在以下条件同时满足时展示：

1. 用户重新进入页面或项目相关 Session，且与上次活动跨越至少一个自然日。
2. 用户存在处于 `open` / `in_progress` / `waiting` 状态的 Project 或 Item。
3. 该项目存在可追溯的最近 Event，且上次简报不是已展示的同一版本。

以下情况不展示：

- 只有 Non Project 或待分类记录。
- Project / Item 已完成或已归档，且没有新的反例。
- 用户仍在当前对话中连续工作，没有发生返回。
- 状态整理失败、来源不可追溯或该简报已失效。

## 4. 简报内容

简报使用已有 Project、Item 和 Event 生成，默认只保留一个核心信息：**每个项目一句话进度摘要**。

摘要按项目中的事项说明“已完成、进行中、尚未开始；确有记录时才说明被某事项阻塞”，控制在一到两行内，不新增“待确认”“为什么现在”或其他解释区块。项目名、最近更新时间和来源 Session 作为轻量元信息显示在摘要旁边。

摘要只能依据已持久化的状态和事件，不自行创造责任人、日期、交付物或下一步；如果来源不足以形成稳定摘要，则不展示该项目的恢复卡片。

## 5. 用户交互

1. 用户重新进入时，在聊天页面的项目区域显示一个名为“Recent Work”的容器，不遮挡主对话；技术能力仍称为 Resume Brief。
2. 默认展示最多 3 个未完成项目，每个项目一行摘要；其余项目折叠为“还有 N 个”，点击后打开全部恢复简报。
3. 当前页面关联的项目优先，其余项目按 `in_progress`、`waiting`、`open` 排序；相同状态下最近更新的在前，已完成或已归档项目不进入恢复简报。
4. 项目名称或“查看项目”入口可点击，打开该 Project 的详情面板；关闭后回到原来的对话位置。
5. 每个项目同时提供“回到 Session”或“继续当前事项”入口，定位到该项目最近相关的目标 Session 和最后一条有效 QA；Session 不存在时，退回项目详情，不自动发送用户消息。
6. 右上角同时提供“收起”和“× 关闭”：收起后保留轻量入口，关闭后在本次浏览器访问中隐藏整个 Recent Work；两种操作都不修改 Project、Item 或 Event。
7. 用户发送新问题后，Recent Work 自动收起，以新 QA 继续维护状态。

## 6. 状态与证据边界

- 页面访问不是工作事实，不创建 Event。
- 简报只读展示 Project / Item / Event 的已持久化内容，不替换 Organizer 的状态更新。
- 冲突、缺失或过时的信息不由简报自行裁决；无法形成稳定摘要时隐藏该项目，保留原始 Project / Item / Event 供用户从详情面板核对。
- 简报不等于 Radar：它可以引导恢复，但不评估新的主动推进动作。

## 7. 3.4 状态断裂实测草案

### 7.1 离开前

> 用户：我们正在推进客户培训门户上线，目标是 8 月 1 日。课程内容和讲师名单已确认，报名页还在测试，上线通知尚未发出。请记录当前项目状态。

> 用户：报名页已通过测试，讲师时间也已确认，但上线通知还没有发出。请写一句简短的内部周报，只说明当前进展。

### 7.2 隔日返回

用户隔日重新打开 Web 页面，但尚未发送新问题。

预期显示：

- 自动显示“客户培训门户上线”恢复进度卡片。
- 用一句话合并两次对话的事实：课程内容、讲师名单和报名页测试已完成，上线通知尚未发送。
- 项目名称可进入 Project 详情；“回到 Session”可定位到最近相关对话。
- 不创建 Event，不生成 Radar，不把通知标记为已发送。

### 7.3 验收标准

- 跨 Session 事实没有丢失、倒序或被混入其他项目。
- 有多个未完成项目时，默认最多展示 3 个，其余可以从“还有 N 个”展开查看。
- 项目详情和来源 Session 两个入口都能从恢复简报到达，且关闭详情后能回到原对话。
- 只读访问不会改变 Project、Item 和 Event 数量或状态。
- 一句话摘要不凭空生成责任人、日期、通知渠道或其他工作步骤。
- 用户点击继续并发送新 QA 后，才按正常 Post-run 链路创建新 Event。
