## 2. 阶段二验收结论

按依赖顺序完成以下任务：

### 2.1 Item 全量重算（已完成）

- Event 纠正归属后，读取新旧 Item 的全部有效 Event。
- 按时间顺序重新计算进展、阻塞、完成、重开和最近 Event。
- `progress_summary` 保留完整 Event 摘要，不再被最后一条 Event 覆盖。
- 新增 Event 没有显式进展补丁时，会在原进展后补入本轮摘要。
- Case 4 对应的失败测试和回归测试已补充。

完成标准：纠正 Event 后，两个受影响 Item 都能得到完整、可追溯的状态。

### 2.2 Project 统计聚合（已完成）

- Organizer 更新 Item，或用户移动 Event / Item 后，重新读取 Project 下全部有效 Item。
- 确定性生成整体 Item 进展和由 blocker 导出的基础风险。
- Project 进展逐项包含必要/可选、状态和 Item 最新进展，不再只使用本轮模型摘要。
- 只有必要 Item 计入必要进度；可选 Item 的 blocker 仍会作为风险展示。
- Case 1 对应的失败测试和回归测试已补充。

完成标准：Project 的统计摘要始终包含 Item 最新有效状态，不停留在较早进展。

### 2.3 状态可靠性验收（已完成）

- [x] 连续多条 Event 的摘要不再被最新 Event 覆盖。
- [x] Event 从 Item A 移到 Item B 后，A、B 及其 Project 都正确重算。
- [x] Item 完成或重新打开后，Project 同步变化。
- [x] Radar 数据库、Service、API、Post-run 与执行相关回归通过。
- [x] 通过浏览器重新执行 Case 4，确认纠正后完整进展正确展示。
- [x] 通过浏览器重新执行 Case 1，确认 Project 吸收全部 Item 最新状态。

完成标准：Case 1 和 Case 4 的已知问题修复，新增回归测试通过，其他 V0 Case 不退化。

最终结论：**阶段二通过。** Case 4 证明 Item 能从全部有效 Event 重算且刷新后不回退；Case 1 证明 Project 能聚合全部有效 Item，阶段、摘要和风险随最新进展更新且刷新后保持。详细证据见 [Case 4 归属纠正实测](../03-测试与评测/案例实测/案例四归属纠正实测.md) 和 [Case 1 跨会话项目实测](../03-测试与评测/案例实测/案例一跨会话项目实测.md)。

2026-07-15 增补：新版单事项 Case 证明“一次完整 QA 维护一个主要 Item”本身没有问题；“品牌素材验收”能正确从 `in_progress` 变为 `completed`。新增缺陷位于 Project 完成守门：代码把“当前已记录的必要 Item 全部完成”当成项目完成充分条件。设计现已调整为 Organizer 维护 Project 的 `status` 与 `current_stage`，代码只做统计和非法转换守门。
