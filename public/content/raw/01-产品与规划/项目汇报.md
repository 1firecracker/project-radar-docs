## 1. 项目动机

Agent 已经能够很好地回答当前问题，但真实工作通常不是一次问答：同一件事会跨越多个 Session 和多个时间点，用户不断补充信息、解除阻塞、完成阶段任务，直到整件事真正结束。

Progress Agent 解决三类工作连续性问题：

- **推进中断**：主 Agent 完成了当前请求，却可能错过状态变化后已经成熟的下一步。
- **状态断裂**：进展散落在不同 Session，用户隔几天回来需要重新梳理。
- **条件遗漏**：已经记录的约束、依赖或风险，没有进入当前判断。

因此，我们不是再做一个聊天记录或项目管理工具，而是增加一个负责持续推进的辅助智能体：它理解事情做到哪里，在合适的时机帮助用户恢复状态、发现机会，并在获得授权后继续完成交付。

一句话定位：**推动下一步，记住状态，补齐判断。**

## 2. 产品架构

Progress Agent 不替代主 Agent。主 Agent 负责当前 QA；Progress Agent 在完整 QA 结束或用户重新返回时介入，把分散对话整理成可持续推进的状态。

![Project Radar V0 三层工作架构](../05-资源/图片/progress-agent-architecture.png)

*图源：[确定性 HTML 架构图](../05-资源/架构图工具/project-radar-architecture.html)。*

整条链路可以概括为：

```text
完整 QA 结束
→ Organizer 记录 Event，并更新 Item / Project
→ Recent Work 帮助返回用户恢复状态
→ Radar 判断是否出现尚未提出但已经成熟的下一步
→ 用户确认后，Action Runner 生成实际产物
→ 结果回写状态，继续下一轮推进
```

| 核心能力 | 作用 | 用户价值 |
|---|---|---|
| Organizer | 把完整 QA 转化为 Event，并维护 Item / Project 状态 | 跨 Session 的进展不再丢失 |
| Recent Work | 用户返回时汇总最近项目与继续入口 | 不必重新翻阅历史对话 |
| Radar + Policy Gate | 在 QA 结束后发现成熟机会，并过滤重复、过期或低价值提醒 | 主动介入但不过度打扰 |
| Action Runner | 用户确认后汇总上下文并调用 AgentV3 执行 | 从“提醒一下”走向实际交付 |

状态底座使用 `Event → Item → Project`：Event 记录每次活动，Item 表示可以判断是否完成的事项，Project 聚合多个相关事项。Radar 可以主动发现，但不能越过用户直接执行。

## 3. Demo

Demo 用一个完整办公故事展示产品价值：用户正在推进客户数据分析平台采购，供应商报价与评估信息分散在不同 Session。最后一份报价到达后，主 Agent 只完成用户当前要求的归档命名；QA 结束后，Radar 从整个事项状态中发现资料已经齐备，主动建议生成选型对比表。用户确认后，Action Runner 汇总跨 Session 上下文并生成可编辑 Excel。

| Demo 环节 | 展示内容 | 说明的问题 |
|---|---|---|
| Part 1：Recent Work | 用户返回页面，看到多个项目的一句话进度和继续入口 | 上次做到哪里 |
| Part 2：Radar | 最后一份报价到齐后，出现生成选型表的主动建议 | 现在还能推进什么 |
| Part 3：Action Runner | 用户确认，Agent 生成包含报价与评估信息的 Excel | 建议能否形成交付 |

## 4. 阶段结论

当前已经形成一个真实可运行的 V0 Demo，能够展示跨 Session 状态积累、Recent Work、Radar 主动判断以及授权后生成 Excel 的完整产品形态。

它证明的是产品方向和用户体验已经可以被讨论与验证，而不是所有生产稳定性问题都已解决。下一阶段应围绕真实用户反馈验证“恢复状态”和“主动推进”是否有价值，同时由研发继续完成稳定性收尾；产品侧暂不横向增加更多功能。
