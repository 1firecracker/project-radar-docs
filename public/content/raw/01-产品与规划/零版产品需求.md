# Progress Agent（持续推进智能体）零版产品需求

> 文档状态：已批准  
> 负责人：bwz  
> 最近更新：2026-07-14  
> 权威范围：V0 产品范围、用户流程、交互规则与验收标准

[返回文档总览](../README.md)

## 1. 目标与成功标准

V0 要证明 Progress Agent 可以把跨 Session 的聊天活动沉淀成可持续追踪的状态，并基于状态完成一次 Radar 主动推进闭环。

成功标准：

- 每个正常完成的 QA 都保存为 Event，并可追溯来源 Session。
- Event 可以归入已有 Item、创建独立 Item、创建明确的多事项 Project，或进入 Non Project / 待分类。
- Item 和 Project 可以保存目标、完成标准、进展和完成状态。
- 用户能从右侧 Radar Tab 查看建议，并打开双栏弹窗查看和纠正自动整理结果。
- Post-run 状态更新后可以生成零条或一条高价值 Radar；没有缺口时保持安静。
- 至少一个 Radar 动作真实调用 AgentV3，并产生草稿、计划、清单或总结。
- 五分钟内可以讲清楚“记录 → 整理 → 状态 → 建议 → 执行 → 反馈”的价值。

## 2. QA 完成链路与触发边界

1. 用户与主 Agent 完成一个 QA。
2. 主 Agent 写入 `end_stream`，页面可以立即恢复交互，Post-run 触发后台链路。
3. Organizer 异步创建 Event 壳记录，并记录 QA 与来源 Session。
4. 系统把 Event 归入已有 Item，或创建独立 Item；只有明确的长期多事项目标才创建 Project。
5. 一次性问答进入 Non Project；无法可靠整理的 Event 进入待分类。
6. 系统更新 Item State，并在有关联 Project 时聚合 Project State。
7. 状态更新成功后，Radar Analyzer 检查当前是否存在用户尚未提出的关键缺口。
8. Policy Gate 决定保持安静或保存一条 Radar Item。
9. 前端有限轮询并更新右侧 Radar Tab、双栏弹窗或 Action Bar。
10. 用户可以纠正整理结果，或接受 Radar 让 AgentV3 在目标 Session 执行。

Post-run 是触发时机，不是一项独立产品能力；它在 V0 中依次调用 Organizer 和可选的 Radar 判断。触发边界：

- Post-run 是 V0 唯一的 QA 完成触发时机；Organizer 和 Radar 不在主回答过程中运行。
- 每个正常完成的 QA 都允许创建 Event；Radar 结果允许为空。
- Radar Action Run 仍要创建 Event 并更新状态，但不得递归生成新的 Radar。
- Re-entry 只读取已有 Project、Item、Event 和 Radar，不执行 Organizer 或新的 Radar 判断。
- Resume Brief、Session End、定时扫描和夜间全局重组不属于 V0。
- 流式输出、等待用户回答和用户输入期间不展示新的 Action Bar。

## 3. V0 功能范围

### 3.1 自动整理与状态

- 每个完整 QA 产生一个 Event，一句话描述本轮发生了什么，并保存来源 Session 和消息标识。
- 一个 Event 最多归属一个 Item。
- V0 每个完整 QA 只创建或更新一个主要 Item；同一句话拆分、创建或批量更新多个 Item 留到后续版本。
- Item 可以独立存在，也可以属于一个 Project。
- 默认先创建独立 Item；只有明确包含多个可分事项的长期目标才创建 Project，并先记录当前主要 Item，后续 QA 再增量增加其他 Item。
- Project 可以包含多个必要或可选 Item；新 Item 默认是必要事项。
- 普通问答进入 Non Project，可分为生活、学习、其他等类别。
- 整理失败的 Event 进入待分类，不更新 Item / Project 状态，也不生成 Radar。
- V0 允许用户纠正 Event 归属、移动 Item、确认完成和重新打开。

Item 与 Project 都维护目标、完成标准、状态和进展。完成标准允许初始为空；为空时不能仅靠 Agent 推断进入待完成确认。

用户明确表示整个 Item 或 Project 已完成时，可以直接标记完成；只完成某一步时只更新进展。Agent 推断完成时只能进入 `completion_candidate`。后续出现明确反例时允许重新打开，并保留原完成 Event。

Item 与 Project 使用不同层级的完成证据：某个 Item 的 `completion_confirmed` 只能完成该 Item。Project 不能因为“当前已记录的必要 Item 全部完成”自动进入 `completion_candidate`；还必须有覆盖整个 Project 完成标准的项目级证据。若本轮只确认一个 Item 完成，Project 保持 `open`、`in_progress` 或 `waiting`，并更新当前阶段和进展。

### 3.2 Radar 侧栏与项目事项弹窗

- 右侧现有工作区提供 `Todolist / Radar` 双 Tab；有待处理建议时 Radar Tab 显示数量徽标。
- Radar Tab 展示当前最高价值建议、主要动作、稍后和忽略，并提供“查看全部项目”。
- “查看全部项目”打开居中双栏弹窗，不跳转到独立页面。
- 左侧展示 Project、独立 Item、Non Project 和待分类。
- Project 条目展示目标、状态和必要 Item 完成进度。
- 右侧展示所选 Project 的 Item，或所选 Item 的目标、完成标准、进展、阻塞、近期 Event 和来源 Session。
- 支持把 Event 移到其他 Item、改为 Non Project、移动 Item、标记完成和重新打开。
- V0 不支持手工创建、删除、合并 Project / Item，也不支持批量整理。

### 3.3 Radar 生成与展示

- 状态整理成功后最多形成一条可展示 Radar Item。
- Radar 必须基于一个明确 Item 或 Project，并包含建议动作和至少一条可追溯证据。
- 无法确定目标、执行 Session 或动作时不生成。
- Radar 只允许基于目标 Session 的最新完整 QA；用户开始下一轮真实对话后，上一轮尚未接受的建议自动过期，不再打断用户。
- 如果主 Agent 的最终回答已经指出、警告或主动提出处理同一缺口，Radar 不重复提醒。
- 同一时刻最多展示一条最高价值建议。
- 强相关时在输入框上方显示 Action Bar；详情展示原因、证据和建议动作。
- Re-entry 只恢复仍有效的待处理 Radar；V0 不生成跨天恢复简报。

### 3.4 推进执行与反馈

- 点击主要动作后，聊天区展示一条简短、可理解的用户可见消息。
- 后端使用 Project、Item、相关 Event 和目标 Session 构造隐藏 Context Bundle。
- 接受接口先锁定 Radar，再携带 `radar_item_id` 调用现有 AgentV3 SSE 链路。
- Radar Action Run 完成后更新原 Radar 状态，同时生成 Event 并更新关联 Item / Project 状态。
- V0 支持“有帮助”“没帮助”“稍后提醒”和“忽略”四类最小反馈；更细的频率、完成和上下文原因留到 V1。

## 4. 交互与状态

### 4.1 Action Bar 与详情

示例：

```text
Project Radar
“设计 Project State”已形成数据模型，但完成标准尚未落入技术文档。
[让 Agent 补齐设计] [查看依据] [稍后]
```

展示条件：

- Radar 关联的 Item / Project 仍有效。
- 当前 Session 与 `target_session_id` 匹配。
- Radar 状态为等待处理并包含明确动作。
- 主 Agent 已结束，输入框为空，用户未处于输入状态。

不展示条件：

- 主 Agent 正在流式输出或等待用户回答。
- Radar 来源已不是当前 Session 的最新真实 QA，或已被新一轮对话取代。
- Event 仍在待分类或状态更新失败。
- 当前 Session 与执行目标不匹配。
- 建议只有信息价值，没有下一步动作。

### 4.2 Item 与 Project 状态

```text
open → in_progress → waiting → completion_candidate → completed → archived
completed → in_progress
```

| 状态 | 含义 |
|---|---|
| open | 已创建，尚未开始明显推进 |
| in_progress | 已有推进记录，仍需继续 |
| waiting | 等待用户、外部条件或其他事项 |
| completion_candidate | Agent 判断可能完成，等待确认 |
| completed | 用户明确确认或完成证据已确认 |
| archived | 不再主动跟进的历史事项或项目 |

Project 进入 `completion_candidate` 需要同时满足：所有已声明的必要 Item 已完成、没有阻止目标完成的 blocker，并且存在覆盖整个 Project 完成标准的项目级证据。仅有“当前数据库中的必要 Item 全部完成”是必要条件，不是充分条件；完成标准只是非空字符串也不代表已经满足。用户明确确认整个 Project 完成时可以直接进入 `completed`。

### 4.3 Radar 状态

```text
pending → accepted → running → done
pending → snoozed
pending → ignored
pending → superseded
snoozed → superseded
running → failed
```

Radar 的 `done` 只表示建议动作已经执行完成，不等同于关联 Item 或 Project 已完成。
`superseded` 表示用户已开始更新的真实对话，旧建议由系统淘汰；它不是用户主动选择的 `ignored`。已经进入 `accepted` 或 `running` 的动作已获得用户授权，不会被新对话自动中断。

## 5. 验收标准

### 5.1 整理与追踪

- 正常完成一个 QA 后，只产生一条对应 Event；重复调度不会重复创建。
- Event 保存来源 Session，并能从弹窗跳转或定位到来源。
- V0 一轮只创建或更新一个主要 Item；Project 的其他 Item 随后续 QA 增量形成。
- 连续推进同一事项的 QA 可以追加到同一 Item。
- 新的单一目标可以创建独立 Item；明确的多事项目标可以创建 Project。
- 一次性问答进入 Non Project；整理失败进入待分类。
- 用户纠正 Event 或移动 Item 后，相关 Item / Project 状态重新计算。
- 用户明确确认完成时更新状态；完成事项可以因后续证据重新打开。
- Item 完成但 Project 完成标准尚未满足时，Project 保持未完成状态，不得自动进入 `completion_candidate`。

### 5.2 展示与纠正

- 右侧 Radar Tab 可以展示当前建议，并通过“查看全部项目”打开居中双栏弹窗。
- 弹窗可以查看 Project、独立 Item、Non Project 和待分类。
- Project 进度只统计必要 Item；可选 Item 不阻止项目完成。
- 用户可以查看近期 Event 的一句话摘要和来源 Session。
- V0 不出现手工创建、删除、合并和批量管理入口。

### 5.3 Radar 与执行

- 状态更新后可以生成零条或一条 Radar。
- 没有高价值缺口时不生成或展示 Radar。
- Re-entry 不触发 Organizer 或 Radar 生成。
- 点击主要动作后，Radar 依次进入 accepted、running 和 done；启动、执行、终止或意外等待失败时进入 failed。
- AgentV3 至少真实生成一份计划、草稿、清单或总结。
- Radar Action Run 更新状态但不递归生成新的 Radar。
- “有帮助”“没帮助”“稍后提醒”“忽略”四类最小反馈均可提交并持久化。

## 6. 非目标与后续版本

V0 不实现：

- 独立 Dashboard 或全屏项目管理器。
- 手工创建、删除、合并和批量整理 Project / Item。
- 一个 Event 同时归属多个 Item。
- 夜间自动重组、Item 合并拆分和自动归入 Project。
- 跨工作日或周末返回时自动生成 Resume Brief。
- Session End 生成 Radar、多 Radar 信息流和完整偏好学习。
- 外部系统监控、浏览器推送和对话进行中的实时插话。

V0.1 先修复状态聚合、历史重算并提高 Radar 判断与证据质量；V0.2 再增加只读 Resume Brief。V1 增加 Session End、时间扫描、停滞检测、夜间重组、合并拆分和多项目恢复；V2 再考虑外部系统变化和持续监控。
