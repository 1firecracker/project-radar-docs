## 1. 验证目标与测试策略

V0 需要证明以下事实：

- Radar 只在正确项目和会话中出现。
- Radar 不干扰用户当前对话。
- 用户能够理解建议原因和证据。
- 用户接受后，AgentV3 能产生真实结果。
- Radar 状态和反馈可以持久化并恢复。

### 1.1 单元测试

- Radar 生成条件。
- LLM 扫描输出的严格结构校验与空结果降级。
- Action Bar 展示条件。
- 状态转换。
- 去重与最小新鲜度检查。
- Context Bundle 构造。
- Radar Action Run 防递归。

### 1.2 接口测试

- 扫描、查询、接受和反馈接口。
- 用户权限与会话归属。
- 重复接受和非法状态转换。
- 不存在、过期和已完成 Item。

### 1.3 集成测试

- Post-run Scan 写入 SQLite 后，前端能够查询。
- 前端收到 `end_stream` 后按有限轮询策略获取异步结果。
- Re-entry 只查询并恢复已有 Item，不触发扫描或模型推理。
- 接受接口完成状态锁定后，前端携带 `radar_item_id` 调用现有 `/api/query` 和 SSE。
- Context Bundle 以隐藏 `functional_instruction` 注入，不显示成普通用户消息。
- Agent 成功或失败后状态正确持久化。

### 1.4 人工端到端测试

- 页面展示位置和低打扰体验。
- 详情信息是否足够解释建议。
- 五分钟演示是否稳定、可理解、可重复。

## 2. 核心测试用例

| 编号 | 场景 | 预期结果 |
|---|---|---|
| 雷达-001 | 项目主会话存在一条待处理建议 | 显示右侧入口和 Action Bar |
| 雷达-002 | 打开无关会话 | 不显示 Action Bar |
| 雷达-003 | 主 Agent 正在流式输出 | 不插入新的 Action Bar |
| 雷达-004 | 系统等待用户回答 | 不显示新的 Radar |
| 雷达-005 | 用户正在输入 | Radar 不突然淡入 |
| 雷达-006 | Radar 没有明确动作 | 不生成或不展示 |
| 雷达-007 | 用户查看依据 | 展示原因、目标会话和至少一条证据 |
| 雷达-008 | 用户接受建议 | 状态依次进入 accepted、running、done |
| 雷达-009 | 用户连续点击接受 | 只启动一个 Agent Run |
| 雷达-010 | Agent 执行失败 | Item 进入 failed，并允许重试 |
| 雷达-011 | 用户点击稍后 | Item 进入 snoozed，冷却期内不展示 |
| 雷达-012 | 用户点击忽略 | Item 进入 ignored，不再显示 |
| 雷达-013 | 用户提交五类反馈 | 反馈写入数据库，当前状态正确更新 |
| 雷达-014 | 相同去重键已经完成 | 不生成重复 Item |
| 雷达-015 | 目标会话已删除 | 新鲜度检查失败，不执行 |
| 雷达-016 | 页面刷新或重新进入 | 可以恢复仍有效的待处理 Item |
| 雷达-017 | 其他用户请求 Item | 返回无权限或不存在，不泄露内容 |
| 雷达-018 | 点击推进 | AgentV3 生成真实草稿、计划或清单 |
| 雷达-019 | Re-entry 时不存在待处理 Item | 返回空列表，不触发扫描，不生成新 Item |
| 雷达-020 | 会话进入 Session End | V0 不创建 Radar Item |
| 雷达-021 | `end_stream` 后扫描尚未完成 | 前端最多查询六次，命中后停止，持续为空时安静退出 |
| 雷达-022 | 接受 Radar 后执行 | accept 只锁定状态，随后 `/api/query` 通过原 SSE 返回 Agent 结果 |
| 雷达-023 | Radar Action Run 完成 | 更新原 Item，但不触发新的 Post-run Scan |
| 雷达-024 | 未登录用户访问 Radar API | 返回 401，不读取共享匿名用户数据 |
| 雷达-025 | LLM 返回非法结构或抛出异常 | 不创建 Item，主 Agent 回复与页面状态不受影响 |

## 3. Demo 数据

### 3.1 项目

- 项目名称：Proactive Agent Demo。
- 项目目标：确定 Project Radar 的 V0 产品范围并准备演示。
- 主工作会话：包含最近产品决策的会话。
- 来源会话：产品动机讨论、交互形态讨论和 V0 范围讨论。

### 3.2 初始 Radar Item

```text
类型：决策
标题：收敛 Proactive Agent V0 范围
摘要：产品形态已经明确，但第一版范围尚未锁定。
为什么现在：继续扩展形态会降低 Demo 聚焦度。
为什么在当前会话：当前会话包含最近的产品决策上下文。
建议动作：让 Agent 生成 V0 范围。
预期产物：范围清单。
```

证据至少包含：

- 已经完成产品形态讨论。
- 多次明确目标是产品演示。
- 第一版需要区分真实能力与模拟能力。

## 4. 五分钟演示脚本

### 4.1 正常对话

用户输入：

```text
帮我整理一下 proactive agent demo 的产品形态。
```

主 Agent 正常回答。讲解重点：当前任务不被 Radar 打断。

### 4.2 Radar 出现

主回复结束后，右侧出现一个推进建议，输入框上方显示 Action Bar。

讲解重点：Radar 在任务边界扫描，而不是在用户说话时插话。

### 4.3 查看依据

用户点击“查看依据”。详情展示为什么现在、为什么在当前会话，以及来源会话、记忆或待办证据。

讲解重点：建议可解释，并且有明确项目与会话归属。

### 4.4 让 Agent 推进

用户点击“让 Agent 生成 V0 范围”。页面显示简短的 Radar 触发消息，AgentV3 根据 Context Bundle 生成范围清单。

讲解重点：这不是普通通知，接受后会真实产出下一步工作成果。

### 4.5 完成反馈

Radar 状态进入完成，用户点击“有用”。刷新或重新进入会话后，已完成 Item 不再作为待处理建议出现。

讲解重点：发现、解释、行动、产出和反馈形成完整闭环。

## 5. 演示准备与验收

### 5.1 演示前检查

- [ ] 后端和前端可以正常启动。
- [ ] 演示用户和项目数据已经初始化。
- [ ] 主会话和来源会话存在，并已写入 `project_id`、`project_role` 和 `project_brief`。
- [ ] 初始 Radar Item 或扫描输入可以稳定生成。
- [ ] AgentV3 使用的模型和配置可用。
- [ ] 演示配置已关闭 next questions。
- [ ] 接受建议后可以生成目标产物。
- [ ] 数据库中没有残留的同去重键完成记录。
- [ ] 页面处于正确项目主会话。

### 5.2 演示失败兜底

- 扫描未生成时，使用预置 Radar Item 继续演示。
- Agent 执行失败时，展示失败状态与重试能力，并切换到预先保存的产出示例。
- 网络或模型不可用时，只能作为界面演示，不应宣称真实执行链路已经成功。
- 演示前保留一份干净的数据库快照，便于恢复初始状态。

### 5.3 最终验收清单

- [ ] 正确会话能够显示单条 Radar。
- [ ] 无关会话、流式输出和输入状态不会被打扰。
- [ ] 用户能够查看原因和证据。
- [ ] 接受建议后真实调用 AgentV3。
- [ ] 异步扫描通过有限轮询展示，不延迟主 Agent 的 `end_stream`。
- [ ] Radar Action Run 不会递归生成新 Radar。
- [ ] 状态流转与持久化正确。
- [ ] 五类反馈能够保存。
- [ ] 重新进入后不会重复显示已处理建议。
- [ ] 重新进入时不会触发新的 Radar 扫描。
- [ ] Session End 不会在 V0 中生成 Radar。
- [ ] 五分钟演示可以连续运行两次且结果一致。
