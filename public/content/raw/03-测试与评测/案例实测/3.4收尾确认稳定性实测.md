## 4. 第二次修订与执行结果

保留 Session A 的 blocker 设计。Session B 在提供最终 PDF 文件名和页数后，不再要求检查文件名，而是要求润色一句内部通知：

> 继续“完成季度经营报告归档”这个事项。最终 PDF 已经生成，共 48 页，文件名确定为“季度经营报告_2026Q2_最终版.pdf”。请帮我把这句话润色成一句简短的内部通知：“季度经营报告已经生成，大家可以查阅。”

这仍是正常的 Web 办公请求，同时将主 Agent 的当前交付限制为内部通知。最终核对单没有被用户提出，且 blocker 已经解除，适合由 Radar 独立发现。

使用账号 `radar34_v3_r1_0716` 执行后：

- Session A 仍正确识别 blocker 并保持 Radar 安静。
- Session B 主 Agent 生成了内部通知，并以“需要我帮你起草核对单吗”形成泛化邀约。
- 决策审计把该句识别为 `coverage.kind=generic_offer`；按统一规则，泛化邀约没有实际产出核对单，不算动作已覆盖。
- Radar 正确生成“起草季度经营报告最终核对单”，执行后产物保留“核对单待确认”和“李明待签字”，Item 仍为 `in_progress`。
- 但主 Agent 在通知中补写“现已归档”和“已完成并归档”，与归档尚未完成的事实冲突。

## 5. 第三次修订

Session B 增加真实业务约束：整个归档还没有完成，通知不得写成“已归档”。最终脚本为：

> 继续“完成季度经营报告归档”这个事项。最终 PDF 已经生成，共 48 页，文件名确定为“季度经营报告_2026Q2_最终版.pdf”，但整个归档还没有完成。请把“季度经营报告最终 PDF 已经生成，大家可以查阅”润色成一句简短的内部通知，不要写成“已归档”。

该修改只约束通知事实准确性，不改变 Radar 机会、blocker 或目标动作。连续通过次数清零，下一轮必须从新账号重新执行。
