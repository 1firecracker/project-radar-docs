### 2.2 第 0 步：建立干净基线

#### 2.1 为什么删除对话后仍能看到旧项目

聊天记录与 Project Radar 状态是两套独立持久化数据。删除聊天只清理 `conversations` 及其消息、附件，不会自动删除跨 Session 的 `projects`、`tracked_items`、`activity_events`、`radar_items` 和 `radar_feedback`。

因此，“删除旧对话”后仍在“查看全部项目”中看到旧 Project，不是浏览器缓存，而是 Project Radar 按设计保留了项目状态。

本次为了重新执行测试，保留 `bwz` 账号、登录 Session 和模型配置，只清空该用户的业务数据。清理后数据库六类数据均为 0，刷新页面显示“暂无对话历史”。

![清理后的空白状态](../../05-资源/图片/Case1实测/00-清理后的空白状态.jpg)

#### 2.2 后续应该怎样提供清理能力

普通用户删除一个 Session 时，不应默认连带删除跨 Session Project。为了测试和演示，建议后续增加一个独立的“重置 Project Radar 测试数据”入口，并要求二次确认。它应明确区分：

- 只删除聊天记录；
- 只清理 Project Radar 状态；
- 同时清理两者并恢复为空白测试账号。

### 2.7 实测发现与后续建议

#### 7.1 必须修正：Project 聚合摘要没有随 Item 更新

Session B 新增 Item 后，Project 的必需进度从 `0/1` 变为 `0/2`，但 `current_stage` 和 `progress_summary` 仍是首轮内容。Project State 应在 Item 新增或更新后重新聚合，否则跨 Session 总览会落后于 Item 事实。

#### 7.2 建议优化：明确 Post-run 的完成反馈

主 Agent 回答结束、Organizer 完成和 Radar Analyzer 完成之间存在短暂时间差。当前界面会先显示空 Radar，随后才出现建议。建议增加“正在整理项目进展”或“正在分析下一步”的状态，避免演示和用户使用时误判。

#### 7.3 建议优化：提供测试数据重置入口

删除对话不应自动删除长期项目，这是正确的产品语义；但研发和演示需要可控的清理方式。建议增加仅开发/测试环境可见的重置入口，并把聊天、Project Radar 和全部业务数据三种范围明确分开。

#### 7.4 文档上传能力不属于本 Case 的通过标准

聊天输入区存在附件入口，但当前 Project、Item 和 Event 详情没有项目文档字段。本次重跑只验证对话文本形成跨 Session 项目状态，没有把上传文件作为 Project 资料进行验证。若要验证文档能力，应单独增加“上传发布说明并检查文档证据归属”的 Case，不能把聊天附件入口等同于项目级文档管理。

### 2.8 Case 1 判定

综合判定：**部分通过，可以展示跨 Session 主链路，但应先修正 Project 聚合摘要。**

已经证明的能力：

- 长期目标能够创建 Project；
- 多轮 QA 能持续更新 Item；
- 不同 Session 能汇入同一 Project；
- Event 保留来源 Session；
- Radar 能过滤旧建议，并在最新 Session 提出具有明确前置价值的决策建议。

尚未达到理想状态的部分：

- Project 聚合状态没有完整吸收最新 Item；
- Post-run 分阶段完成时缺少明确的界面反馈；
- 测试环境缺少安全、可见的 Project Radar 重置能力。
