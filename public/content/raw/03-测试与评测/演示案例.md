## 1. 使用说明

本文提供六个可独立执行的真实模型 Case，用于展示 Progress Agent V0 如何把完整 QA 整理为 Event、Item 和 Project，如何区分 Non Project，以及如何生成、执行和反馈 Radar。

每个 Case 建议使用独立测试用户，或在执行前确认不存在同名 Project、Item 和待处理 Radar。每发送一条用户消息，都要等待主 Agent 完整结束，再等待 Post-run 触发的 Organizer 完成后继续下一条。不要一次粘贴整个 Case，否则无法验证逐轮状态变化。

每轮观察三个结果：

1. **主 Agent 回答**：只判断当前问题是否得到正常回应。
2. **Organizer 整理**：确认 Post-run 触发后，在右侧 Radar 的“查看全部项目”中检查 Event、Item、Project、Non Project 或待分类。
3. **Radar 行为**：检查系统应该保持安静，还是展示一条具有明确动作的建议。

模型可以调整标题和摘要措辞，但不能改变 Case 中规定的核心归属、状态边界和动作价值。若 token、网络或模型不可用，本轮只能记录为环境失败，不能使用预置数据冒充真实整理成功。

## 2. 执行前准备

- 后端和前端均从当前 `codex/bwz` 分支启动。
- `TOKENHUB_API_KEY` 已在启动后端的同一个终端中生效。
- 主聊天、Organizer 和 Radar Analyzer 使用的模型均可访问。
- 演示期间关闭 next questions，避免额外模型调用干扰 Post-run。
- 每轮主回复结束后等待 3～10 秒，再查看 Progress Agent 的项目状态和 Radar。
- 执行下一个 Case 前处理已有 pending Radar，或换用新的测试用户。

## 3. Case 1：跨 Session 创建 Project 并发现发布缺口

> 真实浏览器执行与完整页面截图见：[Case 1 跨会话项目实测](./案例实测/案例一跨会话项目实测.md)。

### 3.1 演示目标

证明一个长期、多事项的目标可以形成 Project，不同 Session 的 QA 可以更新同一 Project，Radar 能基于整体状态指出用户尚未要求处理的发布缺口。

### 3.2 前置消息

#### 第 1 轮：Session A 创建长期目标

用户发送：

> 我计划在 7 月 31 日前发布公司内部 Python SDK v1.0。这个目标至少包含三件事：确定公开 API、完成核心实现、准备发布材料。完成标准是其他团队可以按照文档安装并调用第一个接口。请先帮我把这个目标拆成清晰的工作方向，不需要现在写代码。

预期主 Agent：识别这是一个长期、多事项的发布目标，给出简洁拆分，不直接生成完整发布清单。

预期 Organizer：

- 创建一个类似“发布内部 Python SDK v1.0”的 Project。
- 创建“确定公开 API”“完成核心实现”“准备发布材料”等必要 Item；允许标题不同，但至少包含两个可独立完成的 Item。
- 创建一条来源为 Session A 的 Event。
- Project 状态为 `open` 或 `in_progress`。

预期 Radar：允许保持安静。首轮刚完成目标拆分时，没有必要立即重复主 Agent 已经给出的内容。

#### 第 2 轮：Session A 记录关键决策

用户发送：

> 公开 API 已经确定：第一版只暴露 Client、create_task 和 get_task_status，暂时不开放批量接口。这是 SDK v1.0 的正式范围决策。请帮我检查这个范围是否自洽，但先不要开始实现。

预期主 Agent：检查范围是否自洽，指出明显矛盾即可。

预期 Organizer：

- 新 Event 归入“确定公开 API”Item。
- Event 类型接近 `decision` 或 `progress`。
- Item 进展摘要记录公开 API 范围；若完成标准足够明确，可以进入 `completion_candidate`，但不能无用户确认直接进入 `completed`。

预期 Radar：允许保持安静，不能重复建议“确定 API 范围”。

#### 第 3 轮：Session B 更新同一项目

新建 Session B，用户发送：

> 继续推进之前的“发布内部 Python SDK v1.0”。核心 Client、create_task 和 get_task_status 已经实现，安装说明也写了初稿，并且本地示例可以运行。目前还没有检查版本号、CHANGELOG、构建产物和发布回滚步骤。本轮只帮我总结当前进展，不要替我生成发布清单。

预期主 Agent：总结已经完成和尚未检查的内容，不在当前回答中直接生成发布清单。

预期 Organizer：

- Session B 的 Event 归入 Case 1 已有 Project，而不是创建第二个同名 Project。
- “完成核心实现”Item 更新为 `in_progress` 或 `completion_candidate`。
- “准备发布材料”Item 保持未完成，并记录版本号、CHANGELOG、构建产物、回滚步骤等缺口。
- Project 进展摘要同时体现 Session A 的范围决策和 Session B 的实现进展。

预期 Radar：生成零条或一条发布准备建议。若生成，主要动作应接近“生成 SDK v1.0 发布检查清单”，依据应包含尚未检查的版本号、CHANGELOG、构建产物或回滚步骤，目标 Session 应为 Session B。

### 3.3 演示操作与通过标准

1. 在右侧 Radar Tab 检查待处理数量。
2. 点击“查看全部项目”，确认同一 Project 中同时存在 Session A 和 Session B 的 Event。
3. 若出现 Radar，打开依据，确认它没有建议用户重复完成公开 API 设计。

通过标准：只存在一个 SDK Project；至少两个 Session 的 Event 汇入同一 Project；Radar 动作具体且基于尚未处理的发布缺口。

## 4. Case 2：独立 Item 持续推进

> 真实浏览器执行与完整页面截图见：[Case 2 独立事项实测](./案例实测/案例二独立事项实测.md)。

### 4.1 演示目标

证明单一、可完成的事项默认创建独立 Item，不应为了所有持续事项都创建 Project；Radar 可以针对完成标准中的缺口提出下一步。

### 4.2 前置消息

#### 第 1 轮：创建独立事项

用户发送：

> 我需要在 9 月 1 日前完成护照换发。这是一件独立事项，不属于旅行计划或其他长期项目。完成标准是换发申请成功提交并拿到受理凭证。请帮我确认目标和完成标准是否清楚，不需要展开旅行安排。

预期主 Agent：确认目标、截止时间和完成标准，不扩展成旅行项目。

预期 Organizer：

- 创建一个类似“完成护照换发”的独立 Item。
- `project_id` 为空，不创建 Project。
- Event 来源为当前 Session，Item 状态为 `open` 或 `in_progress`。

预期 Radar：允许保持安静，因为首轮目标和完成标准已经明确。

#### 第 2 轮：更新部分进展并留下明确缺口

用户发送：

> 护照换发这件事有新进展：证件照已经拍好，旧护照也找到了，但我还没有确认完整材料清单，也没有预约办理时间。本轮请只帮我把已完成和未完成的内容总结清楚，不要直接替我生成材料清单。

预期主 Agent：总结现状，不在当前回答中直接生成材料清单。

预期 Organizer：

- 新 Event 追加到原独立 Item。
- Item 状态为 `in_progress`。
- 进展包含“证件照完成、旧护照已找到”。
- 下一步或 blocker 包含“确认材料清单”和“预约办理时间”。

预期 Radar：可生成一条“整理护照换发材料清单”或“生成预约前检查清单”的建议；不能创建旅行 Project，也不能建议制定完整旅行计划。

### 4.3 演示操作与通过标准

1. 在双栏弹窗的 Standalone Items 中找到护照事项。
2. 检查两个 Event 均来自正确 Session。
3. 查看 Radar 的原因和动作，确认建议只服务于护照换发完成标准。

通过标准：始终只有一个独立 Item；没有 Project；Radar 动作与缺失材料或预约直接相关。

## 5. Case 3：Non Project 普通问答保持安静

> 真实浏览器执行与完整页面截图见：[Case 3 非项目问答实测](./案例实测/案例三非项目问答实测.md)。

### 5.1 演示目标

证明每个完整 QA 都会留下 Event，但一次性知识或生活问答不应被强行变成 Item，也不应产生主动推进建议。

### 5.2 前置消息

#### 第 1 轮：一次性学习问答

用户发送：

> 这是一次性知识问答，不需要后续跟踪。请用三句话解释 TLS 握手主要解决什么问题，并说明证书在其中的作用。

预期主 Agent：用三句话完成解释。

预期 Organizer：

- 创建一条 Non Project Event。
- 分类接近 `learning`。
- 不创建 Item 或 Project。

预期 Radar：必须保持安静。

#### 第 2 轮：一次性生活问答

用户发送：

> 这也是一次性问题，不需要记录成待办。新鲜蓝莓放在家用冰箱冷藏通常可以保存多久？请给我一个简短答案。

预期主 Agent：给出简短保存建议。

预期 Organizer：

- 创建第二条 Non Project Event。
- 分类接近 `life`。
- 不创建 Item 或 Project。

预期 Radar：必须保持安静。

### 5.3 演示操作与通过标准

1. 打开双栏弹窗中的 Non Project。
2. 确认两条 Event 都存在，并分别保留来源 Session 和一句话摘要。
3. 确认右侧 Radar Tab 没有建议，输入框上方也没有 Action Bar。

通过标准：两轮都有 Event；没有新增 Item/Project；没有 Radar。分类名称允许存在小范围差异，但不能进入待持续推进事项。

## 6. Case 4：模糊归属与人工纠正

> 真实浏览器执行与完整页面截图见：[Case 4 归属纠正实测](./案例实测/案例四归属纠正实测.md)。

### 6.1 演示目标

证明两个相似事项并存时，系统仍保留 Event 来源和归属理由；用户可以纠正 Event，纠正后相关 Item 状态重新计算。

### 6.2 前置消息

#### 第 1 轮：Session A 创建官网文案事项

用户发送：

> 请持续跟踪一个独立事项：“更新产品官网英文文案”。完成标准是首页、定价页和下载页的英文文案都通过品牌审核。当前首页已经完成，定价页和下载页还没开始。

预期 Organizer：创建独立 Item“更新产品官网英文文案”，状态为 `in_progress`。

#### 第 2 轮：Session B 创建帮助文档事项

新建 Session B，用户发送：

> 请持续跟踪另一个独立事项：“更新产品帮助中心英文文档”。完成标准是安装指南、快速开始和故障排查的英文版全部上线。当前安装指南已经完成，另外两篇还没开始。

预期 Organizer：创建第二个独立 Item，不能与官网文案 Item 合并。

#### 第 3 轮：Session C 输入归属模糊的进展

新建 Session C，用户发送：

> 英文版本今天又完成了一部分：快速开始已经翻译完并通过审核，故障排查还没有处理。请只帮我记录当前进展。

预期主 Agent：总结本轮完成与未完成内容。

预期 Organizer：理想情况应归入“更新产品帮助中心英文文档”，因为“快速开始”和“故障排查”属于帮助中心。若模型误归入官网文案或进入待分类，也属于本 Case 可演示的初始状态。

预期 Radar：纠正前允许保持安静。系统不能基于错误归属立即生成与官网发布相关的高置信建议。

### 6.3 人工纠正步骤

1. 打开“查看全部项目”，进入包含第 3 轮 Event 的 Item 或待分类。
2. 检查 Event 的一句话摘要、来源 Session C 和当前归属。
3. 如果已正确归入帮助中心，为演示纠正能力，可先移动到官网文案并保存，再立即移回帮助中心；讲解时明确这是人工演示操作，不是模型误判。
4. 最终把 Event 归入“更新产品帮助中心英文文档”。
5. 重新打开两个 Item，检查帮助中心进展包含“快速开始完成”，官网文案进展不包含该内容。

### 6.4 通过标准

- Event 的来源 Session 在移动前后保持不变。
- 一个 Event 最终只属于一个 Item。
- 移动后两个受影响 Item 的进展和版本重新计算。
- 用户纠正结果不会因为重新进入页面而丢失。

## 7. Case 5：完成、重新打开与 Radar 执行回流

> 真实浏览器执行与完整页面截图见：[Case 5 完成重开与 Radar 执行实测](./案例实测/案例五完成重开与主动建议执行实测.md)。

### 7.1 演示目标

证明用户可以明确确认完成，后续反例可以重新打开事项；Radar 被接受后真实调用 AgentV3，执行结果形成新 Event，但不会递归生成新的 Radar。

### 7.2 前置消息

#### 第 1 轮：创建带完成标准的事项

用户发送：

> 请持续跟踪一个独立事项：“上线客户反馈表单”。完成标准有三条：公开链接可以访问、必填字段可以成功提交、提交后显示确认提示。先帮我检查这三个完成标准是否足够明确。

预期 Organizer：创建独立 Item，保存三条完成标准，状态为 `open` 或 `in_progress`。

#### 第 2 轮：用户明确确认完成

用户发送：

> 客户反馈表单的公开链接已经可以访问，必填字段提交成功，提交后也会显示确认提示。我已经逐项验证，请把“上线客户反馈表单”标记为完成。

预期主 Agent：确认用户已经明确声明完成。

预期 Organizer：

- 创建 `completion_confirmed` 或含义等价的 Event。
- Item 直接进入 `completed`，不需要停留在 `completion_candidate`。
- 保存完成时间。

预期 Radar：保持安静，不能继续建议完成已经确认的工作。

#### 第 3 轮：Session B 提供明确反例

新建 Session B，用户发送：

> 刚发现一个反例：手机端打开客户反馈表单时，提交按钮会被底部导航遮住，用户实际上无法完成提交。所以这件事还不能算完成。请先记录问题和影响，不要现在写修复代码。

预期主 Agent：确认反例会推翻之前的完成结论，记录影响，不直接写代码。

预期 Organizer：

- 新 Event 归入原 Item。
- Event 类型接近 `reopened` 或 `blocker`。
- Item 从 `completed` 回到 `in_progress` 或 `waiting`。
- 原完成 Event 仍保留在时间线中。

预期 Radar：生成零条或一条补救建议。建议动作应接近“生成移动端修复与回归检查清单”或“制定反馈表单移动端修复计划”，目标 Session 为 Session B。

### 7.3 接受 Radar

若 Radar 出现：

1. 打开依据，确认引用了“手机端提交按钮被遮挡”的 Event。
2. 点击主要动作，让 AgentV3 真实生成修复计划或回归清单。
3. 等待执行完成，检查 Radar 状态从 `pending → accepted → running → done`。
4. 在 Item 时间线中确认新增一条执行结果 Event。
5. 等待一个完整 Post-run 周期，确认这次 Radar Action Run 没有递归生成同类新 Radar。

如果 Radar 没有出现，本 Case 记录为 Analyzer/Policy 未命中，不能手工插入 Radar 后宣称真实闭环成功。

### 7.4 通过标准

- 用户明确确认后 Item 可以进入 `completed`。
- 明确反例出现后 Item 可以重新打开，原完成证据不丢失。
- 接受 Radar 后真实运行 AgentV3，并形成新的 Event。
- Radar 的 `done` 不会自动把 Item 再次标记为完成。
- Radar Action Run 不递归生成新的同类 Radar。

## 8. Case 6：阻塞解除后主动推进

> 真实浏览器执行与完整页面截图见：[Case 6 阻塞解除与主动推进实测](./案例实测/案例六阻塞解除与主动推进实测.md)。

### 8.1 演示目标

证明用户本轮只处理局部归档工作时，Organizer 可以把“材料齐备”更新为跨 Session 项目状态，Radar 再发现用户尚未提出的前端开工机会。这是当前最能体现主动推进发心的 Case。

### 8.2 前置消息

#### 第 1 轮：Session A 建立阻塞状态

用户发送：

> 星河科技新品发布页的页面结构和交互原型已经确认，合同要求7月18日前交付正式页面。但客户尚未提供最终版Logo、品牌色规范、产品主视觉和正式宣传文案，四项材料缺少任意一项都不能开工，因此前端制作目前必须保持暂停，前端开工任务单也尚未制定。请帮我整理当前项目状态。

预期主 Agent：整理当前状态、材料缺口和合同期限，不在材料缺失时制定开工任务单。

预期 Organizer：

- 创建或更新“星河科技新品发布页”Project。
- 创建“前端制作”必要 Item，状态为 `waiting`。
- blocker 包含四项客户材料，进展保留页面结构和原型已确认。

预期 Radar：保持安静。材料未到位时，前端开工尚不可执行。

#### 第 2 轮：Session B 只处理材料归档

新建 Session B，用户发送：

> 客户刚刚确认最终版Logo、品牌色规范、产品主视觉和正式宣传文案已经全部交付，四项材料均验收无误。我要统一归档，请按“星河科技_素材类型_最终版_20260714”的格式列出四个新文件名，方便我复制使用。

预期主 Agent：只输出四个归档文件名，不主动制定前端任务单或排期。

预期 Organizer：

- 新 Event 归入原 Project / Item，而不是创建第二个项目。
- 记录四项材料已交付验收，清除材料 blocker。
- Item 从 `waiting` 更新为 `open` 或 `in_progress`。

预期 Radar：生成“制定前端开工任务单并排期”或含义等价的建议；依据包含四项开工条件刚刚满足和 7 月 18 日合同期限，目标 Session 为 Session B。

如果有限轮询期间尚未展示建议，可以刷新并重新进入 Session B；Re-entry 只能恢复已经生成的 Radar，不能重新触发 Analyzer。

### 8.3 通过标准

- 第一轮材料缺失时 Radar 保持安静。
- 第二轮 Event 跨 Session 更新同一 Project / Item。
- 材料 blocker 被清除，事项不再保持 `waiting`。
- 主 Agent 没有在当前回答中覆盖开工任务单。
- Radar 说明为什么现在值得推进，并提供可由 Agent 直接执行的动作。

## 9. 结果记录模板

每个 Case 执行后记录：

| 字段 | 内容 |
|---|---|
| Case | Case 编号与名称 |
| 测试用户 | 用户名，不记录密码或 token |
| Session | 使用的 Session id 与标题 |
| 实际 Event | 摘要、来源、归属和类型 |
| 实际 Item / Project | 标题、状态和进展 |
| 实际 Radar | 是否出现、标题、原因、动作和目标 Session |
| 执行结果 | 是否接受、Agent 产物、Radar 终态和回流 Event |
| 与预期差异 | 缺失、误归属、状态错误、重复建议或不应出现的 Radar |
| 结论 | 通过、部分通过、失败或环境失败 |

建议执行顺序为 Case 3 → Case 2 → Case 1 → Case 4 → Case 5 → Case 6：先验证安静边界和独立 Item，再逐步增加跨 Session、纠正、执行闭环和真正的主动推进。
