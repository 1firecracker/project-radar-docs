## 1. 演示目标

这次演示不试图介绍所有数据结构和技术模块，只证明一件事：**Agent 不仅能回答当前问题，还能跨对话记住事情做到哪里，在推进条件成熟时主动发现下一步，并在用户授权后完成实际交付。**

整场建议控制在两分钟左右：

| 环节 | 时长 | 目的 |
|---|---:|---|
| 开场 | 15～20 秒 | 说明用户问题和产品定位 |
| 播放 Demo | 81 秒 | 展示 Recent Work、Radar 和执行结果 |
| 收束 | 15～20 秒 | 把三个功能归纳成一条持续推进闭环 |

一句话定位：**Progress Agent 推动下一步，记住状态，补齐判断。**

## 2. 整体故事

用户每天会同时推进多个事项，信息分散在不同 Session。隔几天回来时，他先要重新找回进度；继续工作时，主 Agent 通常只完成眼前请求，不会持续检查整件事是否已经具备下一步条件。

Demo 用一条完整链路说明 Progress Agent 如何介入：

1. 用户返回页面，Recent Work 用项目摘要帮助他快速找回最近工作。
2. 在客户数据分析平台采购事项中，甲、乙报价和评估信息已经分散记录在之前的 Session；最后一份丙方报价到达后，主 Agent 按要求只生成归档文件名。
3. 当前 QA 结束后，Radar 从跨 Session 状态中发现“三家报价和评估信息已经齐备”，主动建议生成选型对比表。
4. 用户确认后，Action Runner 汇总完整上下文，生成可编辑 Excel，并把执行结果继续回写状态。

Recent Work、Radar 和 Action Runner 不是三个孤立功能，而是一件事的三个阶段：**恢复状态 → 发现机会 → 完成交付。**

## 3. 正式讲述稿

### 3.1 开场

> 今天展示的不是一个保存更多聊天记录的功能，而是一个持续推进工作的辅助智能体。现实中，一件事经常跨越多个 Session：用户回来时不知道上次做到哪里，继续交流时，主 Agent 又主要回答眼前问题，容易错过已经成熟的下一步。Progress Agent 希望解决的，就是这种工作连续性问题。

### 3.2 Part 1：Recent Work

> 首先，用户隔几天重新进入页面，不需要逐个翻阅历史对话。Recent Work 会基于已有的 Project、Item 和 Event，给出最近工作的简要状态。用户可以快速看到每件事做到哪里，并进入项目详情或来源 Session 继续工作。

讲解重点：这里展示的是**找回状态**，不是产生新的工作事实，也不是把所有聊天变成项目。

### 3.3 Part 2：Radar 主动提醒

> 接下来进入客户数据分析平台采购事项。此前甲、乙两家的报价和四维评估信息已经记录在不同 Session 中。现在最后一份丙方报价到达，用户只要求主 Agent 为它生成归档文件名，主 Agent 也准确完成了这个当前请求。
>
> 但从整个事项看，三家报价与评估条件已经全部齐备。当前 QA 结束后，Radar 基于跨 Session 的项目状态发现：现在已经适合生成选型对比表，于是主动给出一条可解释、可执行的建议。这就是 Progress Agent 与普通问答的差别——它不打断当前回答，而是在任务边界上检查整件事还能否继续推进。

讲解重点：Radar 不是重复主 Agent。主 Agent 完成的是“丙方报价归档文件名”，Radar 发现的是此前尚未提出的“汇总三家信息并形成选型对比表”。

### 3.4 Part 3：Radar 实际推进

> Progress Agent 不会擅自执行。用户点击确认后，Action Runner 才会把相关 Project、Item、历史 Event 和目标 Session 组织成完整上下文，交给 Agent 执行。
>
> 最终生成的不是一句提醒，而是一份可继续编辑的 Excel：包含三家报价、预算判断和已有的四维评估信息。执行结果还会形成新的 Event，继续更新事项状态，因此这条链路不是一次性的跳转，而是可以持续循环的推进闭环。

### 3.5 收束

> 整个 Demo 展示的是一条完整产品链路：用户回来时快速恢复状态，工作条件成熟时获得主动提醒，确认后直接形成可交付结果。Progress Agent 的核心价值不是替用户做决定，而是在不打扰主任务的前提下，让跨 Session 的工作不再失联，让已经成熟的下一步不再被遗漏。

## 4. 演示包装要求

1. 先讲用户问题，再播放功能；不要从 Post-run、Organizer 或数据库表开始。
2. 视频全屏播放，不展示浏览器标签、终端和调试日志。
3. Part 2 必须让观众看清“用户发送丙方报价—主 Agent 返回文件名—Radar 出现”的先后关系。
4. Part 3 在 Excel 完整加载后停留，指出它已经包含报价、预算和评估信息，而不是只说“生成成功”。
5. 全程统一使用 `Progress Agent` 指代产品；`Radar` 只指主动发现模块，`Recent Work` 只指返回页面时的恢复视图。
6. 明确用户授权边界：Radar 可以主动建议，但不能越过用户直接执行。
7. 说明录像压缩了模型等待时间；页面操作和最终产物来自真实演示链路，不把剪辑包装成实时速度。

## 5. 可宣称范围与答疑

| 问题 | 推荐回答 |
|---|---|
| 这和长期记忆有什么不同？ | 记忆解决“记住什么”；Progress Agent 进一步维护事项状态，并判断什么时候值得推进。 |
| 为什么主 Agent 没直接生成表格？ | 用户当前只要求归档文件名；Radar 从跨 Session 的完整事项状态中发现了另一个已经成熟的动作。 |
| 为什么不自动执行？ | 主动发现和实际授权分离，避免 Agent 在用户未确认时产生外部影响或错误交付。 |
| 这是不是项目管理工具？ | 不是。Project、Item、Event 是帮助 Agent 理解工作连续性的状态底座，V0 不提供完整项目管理器。 |
| 当前版本已经证明了什么？ | 已展示跨 Session 状态累积、Recent Work 恢复、Post-run Radar 判断、用户授权执行和 Excel 结果回流。 |
| 当前版本还没有什么？ | 尚不宣称外部系统持续监控、自动定时推进、夜间重组或完全无人值守执行；这些属于后续版本。 |
