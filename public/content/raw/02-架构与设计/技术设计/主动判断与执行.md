## 1. Radar 生成约束

允许生成 Radar 的条件：

- 关联的 Item / Project 状态已经成功持久化。
- 状态变化后出现了用户尚未提出、已经成熟的推进动作，或当前判断遗漏了已记录的关键条件。
- 建议可以在明确的 `target_session_id` 中执行。
- 建议包含原因、证据和明确动作。
- 关键条件和候选动作可追溯到来源 Event 或明确的 Item / Project 字段；不仅依赖模型生成的摘要。
- 不仅凭领域常识或模型推测新增业务流程、完成标准或交付物。
- 没有未处理的同类建议，且不在冷却期。

每次 Post-run 最多保存一条 Radar。生产 Analyzer 不直接决定 `emit / silent`，而是返回一组可校验的动作级观察：

```json
{
  "opportunity": {
    "action_key": "稳定的动作语义键",
    "action": "一个具体可执行动作",
    "expected_output": "draft | checklist | plan | summary"
  },
  "coverage": {
    "kind": "none | delivered | explicit_recommendation | user_requested | generic_offer | artifact_text",
    "source": "none | assistant_direct | user_request | generated_artifact",
    "evidence": "短证据或 null"
  },
  "constraint_check": {
    "status": "allowed | blocked",
    "reason": "该约束是否命中当前动作"
  },
  "dependency_check": {
    "status": "clear | resolves_current_blocker | blocked_by_current_prerequisite",
    "reason": "该动作与 Item 当前 blocker 的关系"
  },
  "feasibility": "ready | insufficient_evidence | not_actionable",
  "candidate": "动作 allowed 且 ready 时为完整 Candidate，否则为 null"
}
```

Policy Gate 的映射是确定性的：

| 动作级观察 | 最终结果 |
|---|---|
| `ready + allowed + none/generic_offer/artifact_text` | `actionable_gap`，校验并保存 Candidate |
| `ready + allowed + delivered/explicit_recommendation/user_requested` | `main_answer_covered`，保持安静 |
| Item 有 blocker，动作直接解除该 blocker | 允许继续按覆盖、约束和可执行性判断 |
| Item 有 blocker，动作绕过 blocker 进入下游 | `not_actionable`，保持安静 |
| Item 无 blocker，已识别动作且仅有可选信息未定 | 降级为带“待确认”字段的可编辑初稿，`actionable_gap` |
| 无法识别任何可信机会 | `insufficient_evidence` 或 `not_actionable`，保持安静 |
| 无机会、动作被约束阻止或不可推进 | `not_actionable`，保持安静 |

`coverage` 必须证明“同一个动作”已经覆盖，并记录来源。主回答中代用户生成的正文、引用、模拟内容、代码或模板属于 `generated_artifact`；其中出现的建议标记为 `artifact_text`，不能冒充主 Agent 的直接建议。条件式或泛化的继续帮助邀约标记为 `generic_offer`，也不构成覆盖。

`constraint_check` 只检查当前 `opportunity`。例如某项约束阻止最终执行，并不会自动阻止不同的准备、比较或核对动作；禁止范围不能由 Analyzer 自行扩大。约束与可执行性彼此独立：一个动作即使技术上 `ready`，只要被用户明确禁止，Policy Gate 仍静默且 Candidate 必须为空。

`dependency_check` 以 `Item.blockers` 为唯一权威前置条件。存在 blocker 时，V0 只允许直接解除或缓解该 blocker 的动作，不允许用模板、占位表或下游准备绕过等待关系。blocker 为空时，Project `open_questions`、偏好和 Analyzer 临时想到的参数不能被升级为新 blocker；如果机会已明确、未覆盖且未被约束禁止，服务端将 `insufficient_evidence` 降级为基于现有事实的可编辑初稿，未确定内容标为“待确认”，而不是静默或替用户完成最终决策。

对于 `ready + allowed` 的机会，Analyzer 即使判断覆盖成立也要返回 Candidate；Policy Gate 在覆盖成立时丢弃它。这样推送决策由服务端规则产生，而不是再依赖模型同时正确输出一组互相重复的 `decision / reason_code`。

模型静默、模型调用失败、结构非法、Candidate 校验失败、落库失败和来源 QA 过期是不同结果，不能再合并成“没有 Radar”。服务端把最终结果写入 `radar_decisions`；系统分支使用 `invalid_candidate`、`analyzer_error`、`persistence_error`、`stale`，便于按 Event、Run 或 Session 追查。旧的直接 Candidate、空对象和旧决策包仅保留为测试与迁移兼容，生产提示词使用动作级结构化观察协议。

## 2. Context Bundle

点击推进时，后端基于目标 Session 重新构造上下文，包括：

- 用户接受的 Radar 动作。
- Project 目标、完成标准和当前摘要。
- Item 目标、完成标准、进展、下一步和 blocker。
- 相关 Event 摘要与来源 Session。
- Radar 原因、证据和建议动作。
- Radar 生成时冻结的证据与服务端动作信息。

前端只显示一条短消息；完整 Bundle 以 `input_kind="functional_instruction"`、`source="runtime"`、`hidden=true` 注入，前端不能提交或覆盖隐藏内容。
