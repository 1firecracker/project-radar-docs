## 1. 数据模型

### 1.1 Project 与 Item

```ts
type ProjectV0 = {
  id: string; // Project 全局唯一标识。
  user_id: number; // 所属用户，用于权限隔离。
  title: string; // 面向用户展示的项目名称。
  goal?: string; // 项目最终目标；初始上下文不足时允许为空。
  completion_criteria?: string; // 项目完成标准；为空时不能仅靠 Agent 推断完成。
  status: "open" | "in_progress" | "waiting" | "completion_candidate" | "completed" | "archived"; // 项目生命周期状态。
  current_stage?: string; // Organizer 根据项目语义维护的当前阶段。
  progress_summary?: string; // 服务端根据全部有效 Item 确定性汇总的整体进展。
  key_decisions: string[]; // 对后续执行仍有约束力的关键决策。
  risks: string[]; // 当前可能阻止目标完成的风险。
  open_questions: string[]; // 仍需回答或确认的问题。
  version: number; // 乐观并发版本，每次状态更新递增。
  completed_at?: string; // 最近一次进入 completed 的 ISO 8601 时间。
  created_at: string; // 创建时间。
  updated_at: string; // 最近更新时间。
};

type TrackedItemV0 = {
  id: string; // Item 全局唯一标识。
  user_id: number; // 所属用户，用于权限隔离。
  project_id?: string; // 可选 Project；为空表示独立 Item。
  category: "work" | "life" | "learning" | "other"; // 独立 Item 或项目事项的领域分类。
  title: string; // 一句话事项名称。
  goal?: string; // 事项要达到的最终结果。
  completion_criteria?: string; // 判断事项彻底完成的标准。
  status: "open" | "in_progress" | "waiting" | "completion_candidate" | "completed" | "archived"; // 事项生命周期状态。
  required_for_project_completion: boolean; // 是否阻止所属 Project 完成；新建时默认 true。
  progress_summary?: string; // 已经完成内容的一句话摘要。
  next_action?: string; // 当前最合理的下一步。
  blockers: string[]; // 当前阻塞事项继续推进的因素。
  last_event_id?: string; // 最近一次影响状态的 Event 标识。
  version: number; // 乐观并发版本，每次状态更新递增。
  completed_at?: string; // 最近一次进入 completed 的 ISO 8601 时间。
  created_at: string; // 创建时间。
  updated_at: string; // 最近更新时间。
};
```

Project 的 `status` 与 `current_stage` 由 Organizer 在 `project_patch` 中维护，服务端不按 Item 数量自动推进语义状态。Project 进入 `completion_candidate` 的必要条件是所有已声明的必要 Item 完成、没有阻止目标完成的 blocker，并且 Organizer 判断当前 QA 与历史证据已经覆盖整个 Project 完成标准，但用户尚未明确确认结束。`completed` 还要求 `event_type=completion_confirmed` 且 `completion_scope=project`。Item 级完成证据只能完成 Item；后续 Project 级或必要 Item 级反例允许 `completed → in_progress`，原完成 Event 保留。

### 1.2 Activity Event

```ts
type ActivityEventV0 = {
  id: string; // Event 全局唯一标识。
  user_id: number; // 所属用户，用于权限隔离。
  session_id: string; // 产生本次 QA 的来源 Session。
  user_message_id: string; // 本轮用户消息标识。
  assistant_message_id: string; // 本轮最终 Agent 消息标识。
  source_run_id: string; // 本轮 Agent Run 标识，具有唯一约束以保证幂等。
  item_id?: string; // 最多关联一个 Item；Non Project 或待分类时为空。
  summary?: string; // 对本轮 QA 发生内容的一句话描述。
  event_type?: "progress" | "decision" | "blocker" | "completion_confirmed" | "reopened" | "reference"; // 本轮对事项状态的主要影响。
  completion_scope: "none" | "item" | "project"; // 完成或重新打开证据作用层级；普通 Event 为 none。
  category?: "work" | "life" | "learning" | "other"; // Non Project 或独立事项分类。
  processing_status: "processing" | "classified" | "classification_failed"; // 后台整理状态。
  assignment_reason?: string; // 为什么归入当前 Item 或 Non Project。
  last_error?: string; // 最近一次整理失败原因，成功时为空。
  occurred_at: string; // QA 完成时间。
  created_at: string; // Event 壳记录创建时间。
  updated_at: string; // 归属或处理状态最近更新时间。
};
```

Event 的 QA 来源字段不可修改；`item_id`、`category` 与归属原因允许用户纠正。V0 每个 QA 只有一个主要 Event，因此最多创建或更新一个 Item；未来若支持一轮推进多个 Item，需要拆分多个 Event 或引入批量状态补丁。Event 从 Item A 移到 Item B 时，服务端重新计算两个 Item 及其 Project 的受影响状态。

纠正后的 Item 重算读取其全部 `classified` Event，并按 `occurred_at ASC, id DESC` 回放；`progress_summary` 按时间保留全部非空 Event 摘要，`last_event_id` 与公开时间线使用相同的并列时间排序。Item 的目标、完成标准、分类和下一步不是 Event 派生字段，重算时保持不变。

每次关联 Item 更新、Event 移动或 Item 移动后，服务端读取全部非归档 Item，确定性生成 Item 数量、必要/可选进度、`progress_summary` 和由 blocker 导出的基础风险；可选 Item 不阻止完成。Organizer 输出的 Project 级风险与基础风险去重合并。

`status` 与 `current_stage` 不再由 Item 聚合器生成：正常 Post-run 使用 Organizer 的 `project_patch`；人工纠正导致状态重算时，服务端只执行安全降级，不做完成推断。没有项目级完成证据时，即使当前已记录的必要 Item 全部完成，也必须保留未完成状态；原先“完成标准非空 + 当前必要 Item 全部完成即自动进入候选”的规则已经删除。

### 1.3 Radar Item

```ts
type RadarItemV0 = {
  id: string; // Radar Item 全局唯一标识。
  user_id: number; // 所属用户，用于权限隔离。
  tracked_item_id: string; // 建议主要关联的 Tracked Item。
  project_id?: string; // 关联 Project；独立 Item 时为空。
  source_event_ids: string[]; // 支撑建议判断的 Event 标识。
  target_session_id: string; // Action Bar 展示和 Agent 执行的目标 Session。
  type: "risk" | "next_step" | "decision" | "completion"; // 建议类型。
  priority: "high" | "medium" | "low"; // 展示优先级。
  title: string; // 用户可读的短标题。
  summary: string; // 建议内容和潜在影响的一句话概述。
  why_now: string; // 为什么当前任务边界适合提醒。
  why_this_session: string; // 为什么应在目标 Session 展示与执行。
  evidence_refs: Array<{ // 可追溯证据列表，至少一项。
    type: "event" | "session" | "memory" | "todo" | "file"; // 证据来源类型。
    title: string; // 证据的用户可读名称。
    snippet: string; // 支撑判断的精简内容。
    source_id?: string; // 原始 Event、Session、记忆、待办或文件标识。
  }>;
  action_label: string; // 主要操作按钮文案。
  action_prompt: string; // 服务端传给 AgentV3 的执行指令。
  expected_output: "draft" | "checklist" | "plan" | "summary"; // 期望产物类型。
  context_snapshot: { // 生成 Radar 时冻结的服务端上下文。
    project_state?: string; // 关联 Project 当前状态摘要。
    item_state: string; // 关联 Item 当前状态摘要。
    related_event_summaries: string[]; // 相关 Event 的摘要。
    constraints: string[]; // 范围、用户偏好和禁止事项等约束。
  };
  source_item_version: number; // Analyzer 读取的 Item 版本；与当前版本不一致时建议失效。
  source_project_version?: number; // Analyzer 读取的 Project 版本；独立 Item 时为空。
  status: "pending" | "accepted" | "running" | "done" | "snoozed" | "ignored" | "superseded" | "failed"; // Radar 生命周期状态；superseded 表示来源 QA 或依赖状态已更新。
  dedupe_key: string; // 同类建议的稳定去重键。
  execution_run_id?: string; // 接受后关联的 Agent Run 标识。
  snoozed_until?: string; // 稍后处理的恢复时间。
  expires_at?: string; // 超过后不再展示或执行的时间。
  last_error?: string; // 最近一次执行失败原因。
  feedback_type?: "useful" | "not_useful"; // 用户对该 Radar 的一次性价值评价。
  feedback_note?: string; // 可选评价说明。
  feedback_at?: string; // 评价提交时间；为空表示尚未评价。
  created_at: string; // 创建时间。
  updated_at: string; // 最近更新时间。
};
```

Radar 的 `done` 只表示建议动作执行完成，不等同于 Tracked Item 或 Project 已完成。

每条 Radar 最多提交一次 `useful / not_useful` 评价，使用 `feedback_type IS NULL` 的条件更新保证并发下不会重复写入。“稍后提醒”和“忽略”属于 Radar 生命周期操作，只更新 `status` 与 `snoozed_until`，不作为评价记录。V0 不再建立独立 `radar_feedback` 表；如果后续需要保存完整操作历史，应新增语义明确的交互日志，而不是把多次操作混入 Feedback。

### 1.4 Radar Decision

```ts
type RadarDecisionV0 = {
  id: string; // 决策审计记录的全局唯一标识。
  user_id: number; // 所属用户，用于权限隔离。
  source_event_id: string; // 本次判断对应的唯一来源 Event；同一 Event 幂等更新一条记录。
  source_run_id: string; // 来源 Agent Run 标识，便于从 Post-run 日志定位。
  source_session_id: string; // 产生该 QA 的 Session；不是 Radar 最终展示目标。
  outcome: "created" | "existing" | "silent" | "stale" | "invalid" | "failed" | "suppressed"; // 服务端最终结果。
  reason_code: "actionable_gap" | "main_answer_covered" | "insufficient_evidence" | "not_actionable" | "invalid_candidate" | "analyzer_error" | "persistence_error" | "stale"; // 稳定、可统计的决策原因。
  reason: string; // 一句安全、可读的解释；不保存原始提示词、完整模型输出或异常秘密。
  opportunity?: { action_key: string; action: string; expected_output: "draft" | "checklist" | "plan" | "summary" }; // 本轮实际评估的动作。
  coverage?: { kind: string; source: string; evidence?: string }; // 同一动作是否已覆盖以及覆盖来源。
  constraint_check?: { status: "allowed" | "blocked"; reason: string }; // 用户约束是否命中当前动作。
  dependency_check?: { status: "clear" | "resolves_current_blocker" | "blocked_by_current_prerequisite"; reason: string }; // 候选动作是否遵守 Item 当前前置条件。
  feasibility?: "ready" | "insufficient_evidence" | "not_actionable"; // 当前证据是否足以立即产生目标产物。
  radar_item_id?: string; // created / existing 时关联的 Radar Item，其他结果为空。
  created_at: string; // 首次记录时间。
  updated_at: string; // 重复判断覆盖最终结果的时间。
};
```

`RadarDecisionV0` 是内部诊断数据，不直接展示为用户建议。它回答“为什么这轮没有 Radar”，而 `RadarItemV0` 回答“实际向用户推了什么”。

## 2. SQLite 表

V0 复用现有 SQLite，目标结构新增五类表：

- **`projects`**：保存 Project 身份和持久 Project State。
- **`tracked_items`**：保存独立或项目内 Item 及其当前状态。
- **`activity_events`**：保存每个 QA 的来源、摘要、归属、处理状态和 `completion_scope`；新字段默认 `none`，历史 Event 不反推项目级完成。
- **`radar_items`**：保存主动建议、上下文快照、来源 Item / Project 版本、执行状态和一次性评价。
- **`radar_decisions`**：按来源 Event 保存一次 Radar 最终判断，并记录动作、覆盖来源、约束和可执行性；区分产品静默、失效、非法输出和系统失败，成功时关联 `radar_item_id`。

2026-07-22 已确认将一次性评价并入 `radar_items`。当前代码仍保留独立 `radar_feedback` 表，属于迁移前过渡实现；工程需要补充三个评价字段、唯一条件写入和旧数据迁移后再移除该表。

必要索引：

- `activity_events(source_run_id)` 唯一索引：保证每个 QA 只生成一个 Event。
- `activity_events(user_id, item_id, occurred_at)`：按 Item 查询事件时间线。
- `tracked_items(user_id, project_id, status)`：查询 Project 内事项和独立事项。
- `projects(user_id, status, updated_at)`：查询活跃 Project。
- `radar_items(user_id, target_session_id, status)`：查询当前会话的待处理建议。
- `radar_items(user_id, tracked_item_id, dedupe_key, status)`：建议去重。
- `radar_items(execution_run_id)`：Worker 完成时回写状态。
- `radar_decisions(user_id, source_session_id, updated_at)`：按 Session 检查每个 QA 的 Radar 判断。
- `radar_decisions(user_id, source_run_id)`：按 Agent Run 定位唯一判断结果。

## 3. 状态转换

Item / Project：

```text
open → in_progress → waiting → completion_candidate → completed → archived
completed → in_progress
```

Event 整理：

```text
processing → classified
processing → classification_failed
classification_failed → classified
```

Radar：

```text
pending → accepted → running → done
pending → snoozed
pending → ignored
pending → superseded
snoozed → superseded
running → failed
```

`superseded` 是系统终态：表示建议所依据的 QA 已不是目标 Session 的最新真实 QA，或生成时读取的 Item / Project 版本已不是当前版本，不能重新展示、接受或恢复。`ignored` 则表示用户主动忽略，二者必须区分。`accepted/running` 已获得用户授权，不随新 Query 或状态更新自动中断。

Radar Candidate 同时受 QA 新鲜度和状态版本约束：Analyzer 调用前比较来源 Event 的 `user_message_id` 与 Session 最新可见真实 User Record；Analyzer 返回后再次比较；落库事务再比较 `source_item_version`、`source_project_version` 与当前版本。Organizer、普通 Item / Project 更新以及人工纠错只要提交了新版本，都会在同一事务中把依赖旧版本的 `pending/snoozed` Radar 置为 `superseded`，范围可以跨 Session；读取和接受接口还会再次校验版本，防止迟到写入或漏网。Session 为 `running` 或 `waiting_user_input` 时来源默认不新鲜。Organizer 不受这些 Gate 影响，仍完成 Event 和状态更新。

Analyzer 上下文包含最新 QA 的原始用户文本和最终 Assistant 文本，但覆盖判断不再扫描整段文本后直接给出结论。Analyzer 先标识候选动作，再为该动作标注覆盖来源；Policy Gate 只把直接交付、直接推荐或用户明确要求视为覆盖，生成产物内部文字与泛化邀约不算。仅改写标题或动作名称不构成新的主动价值。

用户明确确认当前 Item 成功时，Organizer 写入 `event_type=completion_confirmed`、`completion_scope=item`，只把 Item 标记为 `completed`。用户明确确认整个 Project 成功时，Organizer 使用 `completion_scope=project`；服务端完成硬规则校验后才能把 Project 标记为 `completed`。只完成某一步时写入 `progress` Event；Organizer 推断整个对象可能完成时最多进入 `completion_candidate`。
