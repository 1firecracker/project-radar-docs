# 决策记录：采用任务边界触发

> 决策状态：已采纳  
> 决策日期：2026-07-09  
> 最近更新：2026-07-17  
> 权威范围：Project Radar 的触发时机

[返回文档总览](../../README.md)

## 1. 背景

主动 Agent 如果在对话进行中频繁插话，会打断用户思考和输入。Radar 需要体现主动性，同时避免与主 Agent 争夺注意力。

## 2. 决策与原因

V0 采用任务边界上的 Sidecar 机制：

- Post-run Organizer 是 V0 唯一的状态整理触发器：主 Agent 完成本轮回复或任务后创建 Event，并更新 Item / Project 状态。
- Post-run Scan 是 V0 唯一的 Radar 生成触发器：只有状态整理成功后才检查是否需要生成建议。
- Re-entry 只读取、校验并展示已有的待处理建议，不发起扫描，不生成新建议。
- Session End Scan 属于 V1；V0 不在会话结束时生成 Radar。
- 普通对话进行中不实时插话。

- 主 Agent 可以优先完成用户当前请求。
- 任务边界是更自然的注意力切换时机。
- Post-run 工程入口明确，普通问答和复杂运行可以共用，也最适合把 QA 转成状态，发现状态变化后已经成熟的下一步，并检查已记录关键条件是否遗漏。
- Re-entry 对应用户“现在应该继续什么”的真实需求，但不需要承担第二套生成逻辑。
- V0 不依赖复杂后台任务系统即可跑通闭环。

## 3. 替代方案与影响

### 3.1 每条消息实时扫描并立即展示

主动感更强，但容易打断用户，扫描频率和成本也更高。

### 3.2 只做 Session End Scan

长期体验更自然，但需要可靠定义会话结束、后台调度、过期和重新校验，不适合作为 V0 硬依赖。

### 3.3 固定时间通知

适合长期提醒，但与当前项目上下文和任务边界的关联较弱。

### 3.4 决策影响

- Event 壳记录应先保存；整理或扫描失败不得影响主 Agent 回复，也不能丢失 QA。
- Radar Action Run 仍更新 Event 与状态，但不得递归生成新的 Radar。
- Action Bar 只能在流式输出结束、用户未输入且没有等待回答时出现。
- Re-entry 必须复用已有 Radar 查询和新鲜度检查，不得调用扫描接口。
- V1 再实现真实会话结束整理和预生成，并将其作为独立生成触发器评估。

## 4. 重新评估条件

当 Post-run Scan 的等待影响体验、Session End 信号足够可靠，或用户需要跨天即时看到预生成建议时，应提升 Session End Scan 的优先级。
