# 决策记录：采用嵌入式辅助代理

> 决策状态：已采纳  
> 决策日期：2026-07-09  
> 最近更新：2026-07-10  
> 权威范围：Project Radar 的页面形态

[返回文档总览](../README.md)

## 1. 背景

Project Radar 需要承载项目级主动建议，但 AgentV3 的核心工作区仍然是聊天页面。新增完整项目管理页面会扩大 V0 范围，也容易让 Radar 被理解成通知中心或项目看板。

## 2. 决策与原因

采用嵌入主聊天页的辅助代理形态：

- 右侧 Todolist 区域增加轻量 Radar 入口。
- 输入框上方按需展示一条 Action Bar。
- 详情一次只展示一条建议及其原因、证据和动作。
- 不新增独立 Dashboard。

- 保持用户现有工作流不变。
- 可以在用户准备继续行动的位置直接提供建议。
- 一次一条建议更容易控制打扰和认知负担。
- V0 实现范围更小，更适合验证核心价值。

## 3. 替代方案与影响

### 3.1 独立 Radar Dashboard

能够展示更多项目状态，但会把 V0 变成项目管理产品，偏离“主动推进”的验证目标。

### 3.2 顶部常驻项目状态条

信息可见性高，但长期占据主工作区，且无高价值建议时仍产生视觉噪声。

### 3.3 多卡片信息流

可以容纳更多建议，但容易演变成通知中心，也增加排序、去重和批量管理成本。

### 3.4 决策影响

- V0 同时最多展示一条建议。
- 无可行动事项时页面不显示 Radar Action Bar。
- 完整项目看板不属于 V0 范围。
- 后续即使增加多条管理，也应保留聊天页中的单条主建议。

## 4. 重新评估条件

当用户长期积累大量待处理 Radar、单条入口无法有效管理，或多项目使用成为主要场景时，再评估独立管理页面。
