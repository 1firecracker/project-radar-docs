# 决策记录：跨会话理解并定向执行

> 决策状态：已采纳  
> 决策日期：2026-07-09  
> 最近更新：2026-07-10  
> 权威范围：Radar 的项目归属、展示会话与执行会话

[返回文档总览](../README.md)

## 1. 背景

项目上下文可能分散在多个会话中。如果 Radar 只理解当前会话，会遗漏重要决策和待办；如果它在用户当前打开的任意会话直接执行，又可能使用错误上下文并污染会话历史。

## 2. 决策与原因

长期形态下，Radar 可以跨同一项目的多个会话构造判断，但每条 Item 必须绑定明确目标：

- 当前会话。
- 指定会话。
- 项目主工作会话。
- 新会话。
- 仅查看，不执行。

只有当前会话与目标匹配时才展示 Action Bar。用户接受建议后，系统在目标会话执行，不能默认在任意当前会话推进。

V0 只实现 `target_session_id` 指向一个已有会话；项目主会话通过 `session_metadata.project_role="main"` 人工标记。新建会话和仅查看模式均不进入 V0。

- 跨会话信息是理解长期项目的必要条件。
- 明确目标会话可以避免上下文错乱。
- 项目级判断与会话级执行职责清晰。
- 用户可以理解为什么建议属于当前项目和当前会话。

## 3. 替代方案与影响

### 3.1 只使用当前会话

实现简单，但无法体现 Project Radar 的跨会话价值，也容易遗漏历史决策。

### 3.2 始终在当前打开的会话执行

操作直接，但可能在错误上下文中生成内容，并让会话主题变得混乱。

### 3.3 自动新建会话执行所有建议

隔离性强，但会产生大量会话，增加用户寻找结果的成本。

### 3.4 决策影响

- Radar Item 必须保存项目、来源会话、目标和归属原因。
- 展示前必须检查当前会话是否匹配。
- Context Bundle 由后端基于目标会话重新构造。
- V0 需要配置项目主会话和来源会话。
- 自动项目聚类和主会话识别留到后续版本。

## 4. 重新评估条件

当项目系统能够稳定识别工作空间、任务和文档归属，或产品形成独立项目工作区时，可以调整“主会话”概念，但仍应保留明确执行目标。
