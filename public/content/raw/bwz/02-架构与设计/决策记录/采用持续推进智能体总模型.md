# 决策记录：采用持续推进智能体总模型

> 决策状态：已采纳  
> 决策日期：2026-07-14  
> 最近更新：2026-07-14  
> 权威范围：总产品名称、触发时机、核心能力、状态底座与用户界面的概念层级

[返回文档总览](../../README.md)

## 1. 背景

现有文档使用 Project Radar 指代从对话整理、跨 Session 状态、主动建议到 Agent 执行的整套能力，同时又用 Radar Item 指代一条具体建议。这使总产品、主动建议模块和右侧 Radar 界面的边界不清。

Post-run、Re-entry 和 Session End 也曾与 Organizer、Radar 等能力并列描述，但前者回答“什么时候运行”，后者回答“系统运行什么”。继续沿用这一层级会让恢复进度、时间扫描等后续能力被错误地塞入 Radar。

## 2. 决策

总产品统一命名为 **Progress Agent（持续推进智能体）**：跨对话维护用户事项的真实进度，在合适时机帮助用户恢复上下文、发现关键缺口，并经用户授权继续推进。

Project Radar 保留为 Progress Agent 下的主动发现模块，简称 Radar。它负责基于 Item / Project 状态生成可解释、可执行、可反馈的建议，不再代表整个产品。

统一采用四层模型：

| 层级 | 内容 | 回答的问题 |
|---|---|---|
| 触发时机 | Post-run、Re-entry、未来的 Session End、定时或外部变化 | 什么时候运行 |
| 核心能力 | Organizer、Resume Brief、Radar、Action Runner | 系统做什么 |
| 状态底座 | Event、Item、Project、Radar 与反馈 | 依据什么判断 |
| 用户界面 | 项目总览、恢复简报、Radar 卡片、执行结果 | 用户看到什么 |

具体边界：

- **Post-run** 是一个完整 QA 结束后的触发时机。V0 由它先触发 Organizer，状态更新成功后再触发可选的 Radar 判断。
- **Re-entry** 是用户间隔一段时间返回后的触发时机。V0 只读取 Overview 和有效 Radar；V0.2 再基于相同状态提供只读 Resume Brief。
- **Organizer** 为完整 QA 创建 Event，判断归属并更新 Item / Project。
- **Resume Brief** 回答“上次停在哪里、当前状态如何、从哪里恢复”，不把页面访问伪造成新的工作事实。
- **Radar** 回答“是否出现用户尚未提出、但当前值得推进的动作”。主 Agent 已覆盖相同缺口时保持安静。
- **Action Runner** 在用户授权后构造正确上下文、调用 AgentV3，并把结果作为新 Event 回写状态。
- **Session End、定时扫描和外部变化** 是后续触发器，不属于当前 V0。

## 3. 版本边界

| 版本 | 目标 |
|---|---|
| V0 稳定版 | 稳定现有 Post-run → Organizer → State → Radar → Action Runner → Event 回流闭环 |
| V0.1 | 修复状态聚合与历史重算，提高 Radar 判断、证据和安静能力 |
| V0.2 | 增加只读 Resume Brief，验证用户跨天返回后的进度恢复价值 |
| V1 | 增加时间扫描、停滞检测、夜间重组和多项目恢复 |
| V2 | 感知外部系统变化并形成持续工作雷达 |

Resume Brief 不纳入已经批准的 V0 验收标准；V0.2 实施前必须单独补充产品需求、技术设计和测试用例。

## 4. 文档与实现影响

- 总览、产品概要、V0 需求、技术设计和实施计划使用 Progress Agent 表示总产品。
- `Radar`、`Radar Item`、Radar Tab、Radar API 和 `radar_*` 代码名继续表示主动建议模块，无需为了总产品改名而迁移代码或数据库。
- `Post-run Organizer` 在概念描述中统一写成“由 Post-run 触发的 Organizer”；现有函数和文件名可以保留。
- 既有 Case、截图和专项实施计划属于真实历史证据，不因总产品改名而重写。
- 原有关于任务边界触发、嵌入式辅助代理、跨会话定向执行和 Project → Item → Event 模型的决策继续有效。

## 5. 重新评估条件

当 Progress Agent 的主要价值不再是事项持续推进，或者 Radar、Resume Brief 与状态整理被拆成独立产品时，可以重新评估总名称；无论名称如何变化，触发时机与核心能力仍应保持分层。
