# 决策记录：跨会话理解并定向执行

> 决策状态：已采纳  
> 决策日期：2026-07-09  
> 最近更新：2026-07-15  
> 权威范围：Event 来源、跨 Session 状态、Radar 展示与执行 Session

[返回文档总览](../../README.md)

## 1. 背景

同一个 Item 或 Project 可能在多个 Session 中推进。如果 Radar 只理解当前 Session，会遗漏其他会话中的决策、进展和完成证据；如果它在任意打开的 Session 执行，又可能使用错误上下文并污染会话历史。

## 2. 决策

跨 Session 理解通过 Event 实现，定向执行通过 Radar 的目标 Session 实现：

- 每个 Activity Event 保存 `session_id`、用户消息、Agent 消息和 Run 标识。
- Event 归入 Item 后，Item State 可以聚合来自不同 Session 的进展。
- Organizer 读取跨 Session 的相关 Event 与 Item，维护 Project 的语义状态和当前阶段；服务端只确定性统计 Item 进度与 blocker，因此 Project 不依赖单一项目主会话。
- Radar Item 必须保存 `target_session_id` 和 `why_this_session`。
- V0 默认选择最近且与建议动作直接相关的 Event 来源 Session 作为执行目标。
- 只有当前会话与目标 Session 匹配时才显示 Action Bar；项目与事项弹窗可以在其他会话查看状态，但不能默认在当前会话执行。
- 用户接受后，服务端基于目标 Session、关联 Item / Project 和相关 Event 构造 Context Bundle。

V0 不再依赖 `session_metadata.project_role="main"` 人工指定主会话。Session 只承担来源追溯和执行上下文职责，Project / Item 才承担长期状态职责。

## 3. 替代方案与影响

### 3.1 只使用当前 Session

实现简单，但无法体现跨 Session 价值，也容易遗漏历史决策与完成证据。

### 3.2 始终在当前打开的 Session 执行

操作直接，但可能在错误上下文中生成内容，并让会话主题变得混乱。

### 3.3 固定项目主 Session

能够提供稳定目标，但项目长期推进不一定集中在一个会话，也会重新把 Project 状态绑定到 Session。

### 3.4 决策影响

- Event 必须保存来源 Session，Radar 必须保存目标 Session。
- 展示前必须检查当前会话是否匹配执行目标。
- Context Bundle 只能由服务端构造，不能信任前端提交的归属信息。
- Event 被用户纠正到其他 Item 后，目标 Session 和相关状态需要重新计算。
- V1 可以增加新建 Session 或仅查看模式，但仍应保留明确执行目标。

## 4. 重新评估条件

当系统形成独立项目工作区、支持一个 Event 多归属，或能够在多个 Session 中安全协作执行时，可以调整目标 Session 策略，但不能取消来源追溯和定向执行约束。
