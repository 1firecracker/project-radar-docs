**Goal：** 在现有 AgentV3 中持续完善“QA Event → Item / Project State → Radar → Agent 执行 → 状态回流”的 V0 闭环，并依次推进状态可信、Radar 判断可信和跨时间恢复。

**Architecture：** 主 Agent 写入 `end_stream` 后产生 Post-run 触发，由 Organizer 异步持久化 Event 并更新 Item / Project，再由 Radar Analyzer 与 Policy Gate 决定是否生成建议。前端通过 Overview、纠正接口和有限轮询展示状态，接受 Radar 后由 Action Runner 复用现有 `/api/query` 与 SSE 执行 AgentV3。

**Tech Stack：** React 18、Vite、FastAPI、Python、SQLite、AgentV3。

## 1. 全局约束

- 只修改当前用户有权访问的 Project、Item、Event、Session 和 Radar 数据。
- Post-run 触发的后台能力失败不得延迟或改变主 Agent 的 `end_stream`。
- 每个 `source_run_id` 只能创建一个 Event。
- 一个 Event 最多归属一个 Item；Item 可以不属于 Project。
- Re-entry 与 Session End 不触发 V0 模型整理或 Radar 生成。
- Radar Action Run 更新 Event 与状态，但不递归生成 Radar。
- V0 不增加手工创建、删除、合并、批量整理和夜间重组。

---

## 2. 实施概览

V0 实施原则：

- Post-run 是 V0 唯一的 QA 完成触发时机；主 Agent 写入 `end_stream` 后依次异步运行 Organizer 和可选 Radar 判断。
- 每个正常完成的 QA 只生成一个 Event，整理失败仍保留在待分类。
- Project、Tracked Item 和 Activity Event 使用真实 SQLite 持久化，不再依赖人工 `session_metadata` 项目归属。
- 默认创建独立 Item；只有明确多事项目标才创建 Project。
- Re-entry 只读取已有 Overview 与 Radar，不运行模型。
- Radar Action Run 更新 Event 和状态，但不递归生成 Radar。
- 右侧工作区使用 `Todolist / Radar` 双 Tab；Radar 可打开居中双栏弹窗，V0 不做完整项目管理。
- Context Bundle 必须在服务端构造；所有 API 只支持登录用户。

## 7. 交互续跑的 Post-run 补漏修复

**问题：** 真实用户 QA 如果先进入 `waiting_user_input`，首次 Run 因回答未完成而不执行 Post-run；Todolist 或 ask_user Resume 又被 `control_path="resume"` 排除，导致整轮 QA 没有 Event。`bbb` 的 Case 5 第二轮因此出现“主 Agent 与 Todolist 已完成，但 Project Radar Item 仍为 `in_progress`”。

**目标：** 一个真实用户输入无论经历多少次 Todolist/ask_user Resume，最终正常进入 `idle` 时都使用原始用户消息和最终 Assistant 消息执行且仅执行一次 Post-run。

## 8. Radar 新鲜度与重复提醒抑制

**问题：** Post-run Radar 分析耗时可能跨越下一轮 Query，导致上一轮建议晚到；同时主 Agent 已在最终回答中指出的风险，Radar 可能换个标题再次提醒。

**已实现：**

- [x] 新增系统终态 `superseded`；新真实 Query 在 Conversation claim 的同一事务中淘汰当前 Session 的 `pending/snoozed` Radar。
- [x] `accepted/running` 不自动中断；Radar Action Run 与 Resume 不触发淘汰。
